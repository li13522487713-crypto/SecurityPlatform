# Atlas 复刻 Coze Studio 工作流引擎实施计划

## 背景

当前 Atlas 的 `Atlas.WorkflowCore` 是线性步骤链引擎（WorkflowHost + 顺序 Step），仅有 7 种节点，无 DAG、无流式输出、无版本管理，与 Coze Studio 工作流引擎差距显著。

本计划目标：在保持 C#/.NET 10 + 现有审批流不变的前提下，完全复刻 Coze Studio 工作流引擎能力。

---

## 一、现状与目标对比


| 维度    | 当前 Atlas WorkflowCore | 目标（对齐 Coze）                |
| ----- | --------------------- | -------------------------- |
| 执行模型  | 线性步骤链（Outcomes 链表）    | DAG 图（Nodes + Connections） |
| 节点数量  | 7 种                   | 30+ 种                      |
| 执行模式  | 异步后台队列                | 同步 / SSE 流式 / 异步           |
| 版本管理  | 无                     | Draft → Published，SemVer   |
| 中断恢复  | Suspend/Resume（单种）    | 多类型 Interrupt + Resume     |
| Trace | 无                     | 节点级耗时 + Token 统计           |
| 子流程   | 无                     | SubWorkflow 嵌套             |
| 前端画布  | @antv/x6 基础版          | @xyflow/vue（Vue Flow）      |


审批流引擎保持独立，两套引擎并存。

  - `Enums/WorkflowMode.cs`（Standard / ChatFlow）

  - 新增接口：`IWorkflowCommandService`, `IWorkflowQueryService`

### Phase 2 — DAG 执行引擎 + 同步/异步执行（Sprint 3-4）

**目标：** 能执行含 Entry / Exit / LLM / If / SubWorkflow 的工作流，支持同步 & 异步两种模式。

**核心文件：**

- 新建 `src/backend/Atlas.Infrastructure/Services/WorkflowEngine/`
  - `DagExecutor.cs` — 核心 DAG 执行器（拓扑排序 + 并行分支 + 中断恢复）
  - `NodeExecutionContext.cs` — 节点执行上下文（data bag + cancel token）
  - `INodeExecutor.cs` — 节点执行器接口
  - `NodeExecutorRegistry.cs` — 按 NodeType 注册执行器
- 节点执行器（初版）：
  - `Nodes/EntryNodeExecutor.cs`
  - `Nodes/ExitNodeExecutor.cs`
  - `Nodes/LlmNodeExecutor.cs`（复用现有 `LlmStep` 逻辑）
  - `Nodes/IfNodeExecutor.cs`
  - `Nodes/SubWorkflowNodeExecutor.cs`
- 执行接口更新 `IWorkflowExecutionService`：
  - `SyncRunAsync(id, input)` → 返回输出 + ExecutionRecord
  - `AsyncRunAsync(id, input)` → 返回 ExecutionId
  - `CancelAsync(executionId)`
  - `GetExecutionStatusAsync(executionId)` — 轮询状态
  - `GetWorkflowProcessAsync(executionId)` — 节点列表 + 状态 + 耗时

**API：**

- `POST /api/v1/workflows/{id}/run` — 同步执行
- `POST /api/v1/workflows/{id}/async-run` — 异步执行
- `DELETE /api/v1/executions/{executionId}` — 取消
- `GET /api/v1/executions/{executionId}/process` — 执行进度

### Phase 3 — SSE 流式执行 + 中断恢复 + 单节点调试（Sprint 5-6）

**目标：** 前端可实时看到每个节点的执行状态推送；支持 QuestionAnswer 中断等待用户输入。

**SSE 事件帧格式：**

```
event: node_start
data: {"nodeKey":"llm_1","nodeType":"LLM","nodeTitle":"大模型","executionId":"xxx"}

event: node_complete
data: {"nodeKey":"llm_1","status":"Success","costMs":1200,"output":{"result":"..."}}

event: workflow_done
data: {"status":"Success","totalCostMs":2500}
```

**中断类型：**


| 类型              | 触发节点                       | Resume 数据 |
| --------------- | -------------------------- | --------- |
| `Question`      | QuestionAnswerNodeExecutor | 用户回答文本    |
| `InputReceiver` | InputNodeExecutor          | 表单数据      |


**核心文件：**

- `src/backend/Atlas.WebApi/Controllers/WorkflowStreamController.cs`
  - `GET /api/v1/workflows/{id}/stream-run` — SSE 流式（`text/event-stream`）
  - `POST /api/v1/executions/{execId}/resume` — 提交中断恢复数据
  - `POST /api/v1/workflows/{id}/nodes/{nodeKey}/debug` — 单节点调试
- `src/backend/Atlas.Infrastructure/Services/WorkflowEngine/SseEventEmitter.cs`

## 六、关键技术决策

- **Canvas JSON 格式**：参考 Coze FDL Schema，`nodes[]` + `connections[]`，保存为 JSON 列于数据库
- **DAG 执行**：拓扑排序 + `Task.WhenAll` 并行分支，`CancellationToken` 取消
- **SSE**：`Response.ContentType = "text/event-stream"`，`IAsyncEnumerable<SseEvent>` 推送
- **前端画布**：`@xyflow/vue`（React Flow 的 Vue 3 官方版），自定义 Node/Edge 组件
- **变量传递**：节点间通过 `Dictionary<string, object>` data bag 传递，key = `{nodeKey}.{outputField}`
- **保留 WorkflowCore**：原 `Atlas.WorkflowCore` 中的 Primitives（Delay/While/Foreach 等）可保留作为内部基础，新 DAG 引擎位于 `Atlas.Infrastructure/Services/WorkflowEngine/`
- **保留审批流**：`Atlas.Domain.Approval` 完全不变，两套引擎独立运行
