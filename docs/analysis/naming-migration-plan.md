# 兼容期命名规则与迁移顺序建议（SEC-34）

> 版本：v1.0-draft  
> 产出日期：2026-03-16  
> 任务来源：Linear `SEC-34` / 父任务 `SEC-21`  
> 依赖输入：`docs/analysis/object-mapping-matrix.md`

## 1. 目标与原则

- 为高风险对象定义可执行的兼容期命名规则，避免一次性切换导致中断。
- 给出文档、契约、前端、模型、运行态五类改造波次，明确前置条件与回退策略。
- 迁移排序遵循“先定义、再并行、后切流、最后删除”的低风险路径。

## 2. 兼容期命名规则表

| 对象名 | 旧名称 | 新名称 | 并存策略 | 停用时机 |
|---|---|---|---|---|
| Application（裸用） | Application / app | ApplicationCatalog / TenantApplication / TenantAppInstance | 文档与 API 响应同时输出 `newName` 字段；旧字段保留并标记 `deprecated`。 | 完成 2 个版本窗口且前端全量切换后停用旧字段。 |
| Tenant-App | Tenant-App | TenantApplication | 路由、DTO、文档并存：旧词仅作为别名展示，不再新增旧词字段。 | `v2` 稳定后移除旧词。 |
| LowCodeApp | lowcode-apps | tenant-app-instances | 新增新路由，旧路由转发到新服务并返回 `Deprecation` 响应头。 | 兼容期结束且调用量降至阈值以下。 |
| AppManifest | app-manifests | application-catalogs | 资源名并存；写接口仅允许新资源写入，旧资源只读或转写。 | 新发布链路稳定并完成历史数据回填后。 |
| AiWorkspace / app-workspace | ai-workspace / app-workspace | workspace | 前端菜单先改展示名，路由逐步迁移并保留重定向。 | 重定向命中率连续两周低于阈值后移除旧路由。 |
| Runtime（混用） | runtime / RuntimeRoute | runtime-context / runtime-executions | 将“映射态”和“执行态”拆分为两个资源并行发布。 | 审计与监控全链路切换后停用混用命名。 |
| Project（混用） | project（实体+上下文） | project-asset / project-scope | 请求头先保留 `X-Project-Id`，新增 `X-Project-Scope-Id` 并双写日志。 | 新头覆盖率达标且权限策略验证通过后。 |
| DataSource（混用） | datasource | tenant-data-sources / tenant-application-datasource-bindings | 保留原字段只读；新增绑定资源写入，旧写入入口返回迁移提示。 | 全部写路径切换到绑定资源后。 |

## 3. 改造波次计划表

| 波次编号 | 改造范围 | 前置条件 | 产出物 | 风险等级 |
|---|---|---|---|---|
| Wave 0 | 术语与文档基线 | `SEC-33` 映射表冻结；目标词汇表评审通过。 | 术语字典、命名禁用清单、变更公告模板。 | 低 |
| Wave 1 | 接口契约与 OpenAPI 命名并行 | Wave 0 完成；兼容字段规范确认。 | 新旧字段并存契约、`deprecated` 标记、迁移示例。 | 中 |
| Wave 2 | 前端路由/菜单/服务客户端切换 | Wave 1 发布；前端类型定义完成双栈支持。 | 新路由、兼容重定向、客户端双端点调用策略。 | 高 |
| Wave 3 | 后端领域模型与存储绑定改造 | Wave 2 主路径稳定；数据回填脚本验证通过。 | 新对象模型、绑定关系表、迁移脚本与审计补丁。 | 高 |
| Wave 4 | 运行态与监控收口、旧名下线 | Wave 3 完成；兼容命中率达退役阈值。 | 旧路由/旧字段下线清单、回归报告、发布说明。 | 中 |

## 4. 风险与回退表

| 风险项 | 触发条件 | 影响范围 | 回退方案 |
|---|---|---|---|
| 新旧字段语义不一致 | 前后端对同名字段解释不同，出现脏数据。 | API / 前端 / 文档 | 立即冻结新增字段写入，回退到旧字段单写并补齐映射校验。 |
| 路由切换导致 404 或权限错配 | 前端切换新路由后存在未覆盖页面。 | 前端 / API / 运行态 | 保留旧路由重定向；灰度开关回切旧路由入口。 |
| 数据源绑定迁移造成运行失败 | `DataSourceId` 未正确迁移到绑定关系。 | 数据模型 / 运行态 | 维持双读策略：优先新绑定，缺失时回退旧字段。 |
| 运行态拆分影响监控报表 | runtime 指标按新旧资源分裂，报表缺口。 | 运行态 / 监控 / 审计 | 指标平台做别名聚合；短期保留双口径对照报表。 |
| 兼容期过长导致技术债固化 | 旧接口长期存在且新增需求继续写旧名。 | 文档 / API / 团队协作 | 设置“旧名冻结”规则：兼容期仅修缺陷，禁止新功能落旧接口。 |

## 5. 推荐迁移顺序与关键约束

- 推荐总体顺序：`术语与文档` -> `API 契约并行` -> `前端切换` -> `模型与数据迁移` -> `运行态收口与下线`。
- 最不能乱序执行的 3 个关键点：
  - 必须先冻结术语与映射表，再发布新契约；否则前后端会出现二次命名漂移。
  - 必须先让前端完成双栈读取，再执行后端旧字段下线；否则生产路径会中断。
  - 必须先完成监控与审计别名聚合，再拆 runtime 混用对象；否则无法定位迁移故障。

## 6. 实施协同建议

- `SEC-35`：承接 Wave 1，维护契约级别 `deprecated` 与迁移说明。
- `SEC-37`：承接 Wave 2，执行前端路由、菜单、页面与类型命名整改。
- `SEC-38`：承接 Wave 3~4，执行领域模型、数据迁移、运行态收口与下线。
