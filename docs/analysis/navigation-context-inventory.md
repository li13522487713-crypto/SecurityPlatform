# SEC-40 现有菜单结构与上下文切换点盘点

## 1. 盘点范围与方法

- 盘点对象：顶部导航、侧栏菜单、面包屑、Tenant/App/Project 切换入口。
- 盘点依据：前端布局与路由实现、后端菜单种子数据与动态菜单查询映射。
- 目标：给 SEC-24/SEC-41 提供“当前菜单分组 -> 实际承载层级”的现状地图，不做改名与重构方案设计。

### 2.3 当前菜单分组 -> 实际承载层级（现状图）

|---|---|---|---|
| 平台控制台 | 平台级 | 平台 + 租户资源配置（数据源） | 存在跨层承载 |
| 安全中心 | 平台级 | 平台级 | 基本一致 |
| AI 平台 | 应用平台/创作平台 | 平台 + 应用设计 + 市场运营 + 开放能力 + 调试能力 | 混层明显 |

## 3. 上下文切换点清单

| 切换对象 | 出现位置 | 当前行为 | 是否容易误解 |
|---|---|---|---|
| Tenant | 登录页 `tenantId` 输入框 | 用户登录前手动输入租户 GUID；登录后通过 `X-Tenant-Id` 持久化到本地并自动附带请求头 | 高：用户把“租户输入”理解为账号字段，且登录后缺少显式租户切换器 |
| App（工作台） | 路径参数 `/apps/:appId/*` | 通过 URL 直接进入某个应用工作台；页面头部仅显示 `AppId` tag | 中高：缺少统一 App 下拉切换器，跨应用跳转依赖入口页 |
| App（运行态） | 路径参数 `/r/:appKey/:pageKey` | 运行态菜单由 `appKey` 决定并动态加载 | 中：运行态与工作台是两套入口，用户难判断当前所在域 |
| Project | MainLayout 顶部 `ProjectSwitcher` | 启用项目隔离时显示，默认加载 20 条并支持远程搜索，切换后发出 `project-changed` 事件并写入 `X-Project-Id` | 中：仅 MainLayout 可见，Console/AppWorkspace/Runtime 不一致 |
| 系统配置影响 Project | `ProjectSwitcher` 读取 `getCurrentAppConfig().enableProjectScope` | 后端开关决定是否显示项目切换器 | 中：用户感知为“项目消失/出现”，缺少显式解释 |

## 4. 混层问题表

| 菜单/区域 | 问题类型 | 问题说明 | 影响范围 |
|---|---|---|---|
| `系统管理` 主分组 | 平台混入应用 / 上下文不清 | 同组包含租户、组织、项目、数据源、通知、插件等多层能力，用户难以判断“系统”边界 | 全体管理用户 |
| `AI 平台` 分组 | 应用混入运行 / 调试混入主导航 | 同一分组中并列“应用编辑、插件调试、市场、开放平台、工作台”，任务链路跨角色 | AI 产品、运营、开发、运维用户 |
| `低代码中心` 分组 | 应用混入运行 | “应用/表单设计”与“回写监控”并列，设计态与运行监控混放 | 低代码设计与运维角色 |
| `流程中心` 分组 | 应用混入运行 / 上下文不清 | “我的待办/已办/抄送”（个人运行态）与“流程定义管理/实例管理/设计器”（管理与设计态）并列 | 审批参与人、流程管理员 |
| `ConsoleLayout` 顶部菜单 | 平台混入应用 | 平台级控制台中直接承载租户数据源管理入口，和左侧主菜单重复 | 平台管理员 |
| 多入口并存（`/console/*` 与 `/settings/*`） | 菜单重复 | 同一能力存在控制台入口 + 系统管理入口 + 兼容路由，信息架构重复 | 全体后台用户 |
| Project 切换仅 MainLayout | 上下文不清 | 非 MainLayout 页面（Console/AppWorkspace/Runtime）看不到 Project 状态，导致“为何数据不同”难解释 | 使用项目隔离的所有用户 |
| App 上下文呈现 | 入口过深/过浅 | AppWorkspace 仅显示 AppId，无统一 App 切换；Runtime 通过 `appKey` 独立入口，回跳路径不稳定 | 低代码与运行态用户 |
| 面包屑策略 | 上下文不清 | 面包屑来自当前匹配路由，无法表达 Tenant/App/Project 当前值 | 全体用户 |

## 5. 一页结论摘要

### 5.1 最严重的 5 个问题

1. **系统管理主导航膨胀**：平台、租户、项目、应用配置混在同一分组，边界最不清晰。
2. **AI/低代码/流程三大域均存在“设计态 + 运行态 + 管理态”混放**：角色切换成本高。
3. **上下文切换器分布不一致**：Tenant 仅登录时输入，Project 仅 MainLayout 可见，App 依赖 URL 参数。
4. **控制台与系统菜单重复入口并存**：`/console/*` 与 `/settings/*` 并行导致导航认知负担。
5. **面包屑只反映路径，不反映业务上下文**：用户容易在跨 Tenant/App/Project 时迷失。

### 5.2 最需要优先进入四段式 IA 重构的区域

1. **系统管理（最高优先）**：先拆平台治理、租户治理、项目治理能力。
2. **AI 平台 + 低代码中心（第二优先）**：按“构建态/发布态/运行态”拆分入口，收敛调试入口位置。
3. **流程中心（第三优先）**：分离个人工作台与流程管理后台，避免审批参与人进入管理视图。
4. **全局上下文条（横切优先）**：统一展示 Tenant/App/Project 当前值与切换入口，覆盖所有布局壳层。
