# 构建流程可视化方案

## 一、目标与使用场景

在现实工作中，流程往往跨越多个部门、多个系统，常常让人感到信息不对称、责任不清、改进方向模糊。构建一套清晰的流程可视化方案，就是要把复杂的工作过程用直观的图形和层级化的信息呈现出来，帮助团队在同一个认知起点上对齐目标、把握关键节点、快速发现瓶颈、协同制定改进措施。一个好的可视化方案不仅能回答“现在在做什么、进度如何、资源是否充足、风险点在哪里”，还要能提供“下一步该怎么做、谁来做、多久能看到效果”的清晰路线。围绕这个目标，方案需要覆盖数据来源、信息结构、图表类型、交互方式以及运维与迭代机制等要素，并且要考虑不同角色的需求：管理层需要宏观可控、工作组需要可操作细节、数据岗需要可追溯与治理。

## 二、总体框架与设计原则

可视化方案应当具备清晰的框架和一致的设计语言。核心是把“数据信息知识行动”逐层展开，使每一层都能自洽、可验证。设计原则包括：

- 目标对齐：从业务目标出发，确保每个视图都能映射到具体的业务决策点和改进任务。
- 信息分层：将数据分为策略层、流程层、执行层三类信息，避免一张图同时承载所有信息而致混乱。
- 场景适配：不同使用场景使用不同的视图组合，如管理层看仪表板，执行层看工作流细化。
- 数据可信与治理：建立数据来源、口径、口径变更记录、数据质量检查点，确保可追溯。
- 易读性与可用性：遵循简洁、直观、避免过度设计的原则，做到移动端友好、对新用户友好。
- 持续迭代：以小步试错和快速迭代为常态，通过用户反馈驱动改进。

## 三、信息架构与分层视图

一个清晰的分层视图能让人从全局看到细节，再从细节回到全局。建议的层级结构如下：

- 战略与目标层：包含企业目标、关键绩效指标（KPI）、优先级与资源投入的分配关系。此层以高层仪表板和 KPI 地图为主，强调对齐与优先级。
- 过程与流程层：把核心业务活动按时序整理成流程步骤、里程碑、责任人、输入输出、依赖关系。适合用泳道图、流程图和集成关系图。
- 资源与成本层：展示人力、设备、时间、经费、外部依赖等资源的占用与趋势，帮助发现资源瓶颈和成本异常。
- 风险与合规层：标注潜在风险、控制措施、合规要求、审计轨迹，便于提前预警与整改闭环。
- 数据与指标层：对口径、数据源、刷新频率、数据质量状态进行描述，确保视图背后的数据具备可验证性。

## 四、数据源、口径与治理

可视化的可靠性，源自对数据的严格管理：

- 数据源清单：列出所有数据来源、系统接口、数据提取频率、最近一次刷新时间。
- 口径统一：对关键指标建立统一的定义和计算口径，避免不同视图口径不一致导致的混淆。
- 数据质量检查：设定最小可接受的数据完整性、准确性、及时性阈值，定期执行质量抽检。
- 数据追踪与溯源：每个指标都应有数据源标记、计算逻辑描述、变更记录，方便审计和排错。
- 隐私与合规：对个人或敏感数据做脱敏与权限控制，确保合规性不会被数据可视化破坏。

## 五、可视化组件库与信息呈现

为保持统一性，应建立一套可复用的可视化组件库，覆盖常见的图表类型及其最佳用途：

- 指标卡与仪表盘：快速呈现总体状态和趋势，适合摘要性展示与对比。
- 流程图与泳道图：清晰呈现流程步骤、参与角色、并行与依赖关系，适合流程梳理与瓶颈诊断。
- 甘特图与时间线：展示任务计划、实际进度及里程碑，便于进度控制。
- Sankey/流向图：显示资源流向、投入产出关系，帮助发现资源浪费与效率提升点。
- 数据热点图与热力图：以颜色强度呈现密集度或风险级别，适合大规模监控场景。
- 森林图/树状图：结构化呈现层级信息，如组织、工作包、问题分解。
- 关系网络图：揭示系统之间的耦合与依赖，便于识别关键节点。
- 文本与注释区：为每个视图提供简短的解释、口径说明和可操作建议。

## 六、信息密度与呈现策略

在不同视图中控制信息密度，确保信息传达的效率：

- 核心视图（管理层）尽量简洁，重点突出关键 KPI、风险点、下一步行动。
- 运行视图（执行层、数据岗）提供足够细节，包含输入输出、责任人、时间节拍与异常告警。
- 数据可追溯性视图提供计算公式、数据源、刷新频率和异常处理流程。
- 每千字设置少量关键数据锚点，避免信息过载；图表要配合清晰的标题和简短注释。

## 七、交互设计与用户体验

有效的交互可以提升理解力与决策速度：

- 分层钻取：允许从概览钻取到具体流程步骤、到单项任务的细节信息。
- 过滤与切片：按时间、部门、项目、地区等维度切片数据，快速回答“在什么条件下会怎样？”
- 动态对比：提供“当前 vs 目标”、“计划 vs 实际”的对比视图，便于发现偏差。
- 提示与提醒：在风险阈值被触达时给出清晰的告警与纠偏建议。
- 上下文保护：在深度钻取时保留回溯路径，方便用户随时返回到上层视图。

## 八、实施路线与工作流

建立可视化方案需要分阶段推进，确保落地可控、可持续：

- 第 1 阶段：需求梳理与目标对齐。明确核心用户、关键指标、常见场景和数据源边界。
- 第 2 阶段：信息架构与初步原型。搭建分层视图框架，确立指标口径和数据治理规则，生成初版原型。
- 第 3 阶段：数据接入与质量保障。接入数据源、建立数据清洗流程、实施质量检查。
- 第 4 阶段：界面设计与交互实现。完善组件库、完成主要视图的开发与联动，确保跨设备可用性。
- 第 5 阶段：上线前验证与培训。内部试用、用户体验评估、培训手册与使用指南落地。
- 第 6 阶段：上线后运维与迭代。监控使用情况、收集反馈、定期迭代更新，形成长期治理机制。

## 九、质量控制与评估

确保可视化方案稳定可靠，需要建立多维度的评估与控制：

- 可读性与准确性：单页视图的清晰度和指标口径的一致性要达到预设标准，避免错漏。
- 合规与安全：符合数据治理、隐私保护和行业规范的要求，避免敏感信息暴露。
- 用户接受度：通过用户调研、使用行为分析和可用性测试，评估实际帮助程度。
- 变更可追溯：所有口径调整、数据源变更、视图重构都要有变更记录。
- 质量指标覆盖：设定关键数据点的留存率、完整性、时效性等硬性指标，定期自检。

## 十、场景化应用与案例要点

不同场景下的表达重点不同，应进行场景化设计：

- 项目管控场景：以时间线、资源分配和进度对比为核心，重点展示延期风险与资源瓶颈。
- 运营监控场景：以实时数据、异常告警和趋势预测为重点，突出改进方向和执行效率。
- 质量与合规场景：以过程痕迹、控制点和整改闭环为核心，强调可追溯与审计能力。
- 决策支持场景：以关键洞察、情景模拟和行动方案为主，帮助管理层快速制定决策。

## 十一、风险识别与应对

在落地过程中，可能遇到以下常见挑战及应对策略：

- 数据口径不统一：建立统一口径手册，严格版本控制，变更需多方确认。
- 视图冗杂、信息过载：采用分层展示、优先级排序，核心视图避免信息堆砌。
- 性能与响应慢：对大规模数据进行分片查询、缓存策略、异步加载，确保用户体验。
- 用户采纳度低：通过培训、快速上手模板、工作流示例来提升使用率。
- 安全与合规风险：强化访问控制、日志审计、数据脱敏与最小权限原则。

## 十二、落地的基本模板与路径

为帮助快速落地，可以提供一套简化模板与路径：

- 核心视图模板：仪表板 + 流程图 + 资源分布图的组合，覆盖目标、过程、资源与风险四大维度。
- 指标口径模板：列出关键指标及计算公式、数据源、刷新机制、异常阈值和责任人。
- 数据治理模板：数据源清单、数据质量检查表、溯源路径、变更记录和权限控制清单。
- 交互设计模板：常用交互模式、过滤条件、钻取路径、异常告警规则。
- 上线验收模板：验收清单、用户培训要点、使用指南、后续迭代计划。

通过上述构建与应用，流程可视化不再是单纯的图形展示，而成为推动团队协同、提升执行力的综合能力。它将信息从“散落的点”组织成“连起来的线”，再汇聚成“可执行的行动”，让复杂的工作过程变得清晰、可控、可改善。对企业来说，这样的一套可视化方案不是一次性的工程，而是一个持续进化的能力建设过程，贯穿战略制定、执行落地和持续改进的全生命周期。

## 十三、安全与合规映射（等保 2.0）

围绕等保 2.0 要求，将流程可视化能力与安全控制点建立显式映射，做到“功能可视化、责任可追溯、证据可留存”。

- 身份鉴别：所有流程视图与操作必须基于统一身份认证，支持多因子或证书认证接入；流程视图按用户身份与角色动态裁剪。
- 访问控制：视图、流程定义、实例、数据源与指标均纳入细粒度权限控制，支持租户隔离与最小权限原则。
- 安全审计：记录流程设计、发布、变更、执行、回滚、导出等关键行为，并提供可检索的审计视图。
- 数据完整性：对流程定义、版本、实例日志与关键指标进行完整性校验与防篡改措施。
- 机密性保护：对敏感字段进行脱敏展示、密文存储与传输加密；导出与共享需二次授权。
- 可用性保障：关键视图与指标提供降级展示与缓存兜底，保证在高负载下可用。
- 安全管理制度支撑：流程变更审批、权限变更审批、数据口径变更审批均需纳入流程并可视化留痕。

## 十四、落地到平台的功能清单与页面结构

结合当前系统模块（资产、审批、工作流、告警、审计、身份），建议落地为“可视化中心 + 流程设计器 + 运行态中心 + 治理中心”的整体结构。

### 14.1 功能清单

- 可视化中心：总览仪表板、关键 KPI、流程健康度、瓶颈与风险提示。
- 流程设计器：流程节点拖拽、条件配置、权限配置、版本管理、仿真预览。
- 运行态中心：流程实例列表、节点状态、等待原因、耗时分析、异常重试与补偿。
- 治理中心：口径管理、数据源清单、质量报告、变更记录与审计查询。
- 合规模块：敏感字段管理、脱敏规则、导出审批与水印管理。

### 14.2 页面结构（建议）

- 首页总览：流程健康度、实例趋势、关键告警、待办与风险。
- 流程设计：流程列表、版本对比、流程编辑与预览。
- 流程运行：实例列表、实例详情（节点轨迹、耗时、异常、操作）。
- 资源与成本：人力与资源占用、成本趋势、瓶颈排名。
- 风险与合规：风险地图、控制点覆盖、审计追踪。
- 数据治理：口径管理、数据源、质量检查与修复记录。

## 十五、数据字典与指标口径模板

为保证数据可追溯与跨视图一致性，建议建立统一模板并纳入治理流程。

### 15.1 数据字典模板（示例字段）

- 数据名称：指标或字段名称。
- 业务含义：面向业务的简短解释。
- 数据类型：字符串/数值/布尔/时间等。
- 来源系统：具体系统与表或接口。
- 采集频率：实时/分钟/小时/日。
- 责任人：数据负责人或数据岗。
- 脱敏策略：明文/部分脱敏/完全脱敏。
- 质量要求：完整性、准确性、及时性阈值。
- 变更记录：口径或来源变更的时间与说明。

### 15.2 指标口径模板（示例字段）

- 指标名称：统一命名，避免别名混乱。
- 计算公式：明确计算逻辑与单位。
- 数据范围：时间范围、业务范围、组织范围。
- 口径版本：版本号与生效日期。
- 预警阈值：阈值定义与触发动作。
- 可视化场景：用于哪些视图或报表。
- 审批与发布：口径变更审批流程与发布记录。

## 十六、页面级原型要点与指标清单实例（审批 / 资产 / 告警 / 审计）

围绕核心业务域给出页面级原型要点与指标清单实例，便于落地时形成统一模板与口径。

### 16.1 审批

- 页面：流程总览、流程设计器、实例详情、节点追踪、待办中心、抄送中心。
- 关键视图：流程健康度、节点阻塞热力、审批时长分布、退回/撤回占比、异常终止率。
- 指标示例：平均审批时长、节点超时率、一次通过率、转交率、退回率、并行节点完成率。
- 交互要点：按部门/流程版本/发起人过滤；实例轨迹时间线；节点级耗时与责任人联动。

### 16.2 资产

- 页面：资产总览、资产生命周期、资产变更流、权限申请流、资产风险。
- 关键视图：资产新增/变更趋势、关键资产分布、变更失败率、权限回收时效。
- 指标示例：资产完整率、关键资产覆盖率、变更审批时长、权限回收超时率。
- 交互要点：资产类型/等级过滤；资产与流程实例联动；资产风险与控制点映射。

### 16.3 告警

- 页面：告警总览、处置流程、告警详情、升级链路、处置绩效。
- 关键视图：告警量趋势、处置时长分布、重复告警占比、升级路径耗时。
- 指标示例：MTTA/MTTR、首次响应率、误报率、升级成功率、闭环率。
- 交互要点：严重级别筛选；告警到流程实例回溯；处置节点耗时与瓶颈定位。

### 16.4 审计

- 页面：审计总览、审计轨迹、操作留痕、口径变更、合规检查。
- 关键视图：审计事件趋势、敏感操作分布、未闭环项数量、证据留存率。
- 指标示例：审计覆盖率、敏感操作占比、证据完整率、整改闭环时长。
- 交互要点：按用户/角色/系统过滤；操作链路回放；证据与流程实例关联。

## 十九、实施路线与任务分解（落地执行版）

按照 3 个迭代（每迭代 2 周）推进，兼顾最小可行与可持续扩展。

- 迭代 1：基础可视化与治理底座
  - 建立数据源清单、指标口径库、审计留痕中台能力。
  - 前端落地“可视化中心”骨架：顶部筛选、KPI 卡片、趋势图容器。
  - 后端开放基础查询 API：流程实例列表、指标聚合、审计查询。
  - 安全：统一身份鉴别、细粒度鉴权拦截器、操作审计日志。
- 迭代 2：流程设计与运行态联动
  - 前端接入流程设计器（基于 @antv/x6），实现节点库、画布、属性抽屉、发布校验。
  - 运行态中心：实例详情、轨迹时间线、节点耗时与异常重试。
  - 数据治理：指标阈值预警、质量抽检报表。
  - 安全：发布与删除须二次确认 + 审批；脱敏展示与导出控制。
- 迭代 3：场景化与性能/可用性
  - 场景页：审批/资产/告警/审计的专属组合视图与交互。
  - 资源与成本、风险与合规视图联动；缓存与分片查询优化。
  - 质量评估与用户体验回路：埋点、A/B 或可用性测试，按反馈迭代。

## 二十、技术选型与集成点

- 前端：Vue 3 + Vite + Ant Design Vue，画布选 @antv/x6，图表选 Apache ECharts（或 AntV G2/Graphs）。
- 前后端契约：遵循 `docs/contracts.md` 统一响应模型，分页、错误码复用 Atlas.Core 定义。
- 后端：ASP.NET Core WebAPI，WorkflowCore 扩展已有模块；SqlSugar + SQLite 存储。
- 安全：JWT + 证书双模；租户隔离中间件；NLog + 审计留痕。
- 架构：Clean Architecture，新增前端页面放 `src/frontend/Atlas.WebApp/src/pages/visualization/*`，共享模型放 `src/frontend/Atlas.WebApp/src/types/*`；后端接口在 `Atlas.WebApi` 下新增 `VisualizationController`。

## 二十一、核心 API 草案（示例，遵循统一响应）

- `GET /api/visualization/overview`：返回 KPI、趋势、风险提示（支持 filters: timeRange, department, flowType）。
- `GET /api/visualization/processes`：分页查询流程定义，含版本与发布状态。
- `GET /api/visualization/processes/{id}`：流程定义详情 + 版本列表。
- `GET /api/visualization/instances`：实例列表，支持状态/流程/租户过滤。
- `GET /api/visualization/instances/{id}`：实例轨迹、节点耗时、异常记录。
- `POST /api/visualization/processes/validate`：上传 DSL/画布 JSON，返回校验结果。
- `POST /api/visualization/processes/publish`：发布流程定义（需要审批与审计）。
- `GET /api/visualization/metrics`：多指标聚合（审批/资产/告警/审计场景）。
- `GET /api/visualization/audit`：查询可视化相关操作审计。

## 二十二、操作权限矩阵（角色 × 关键操作）

- 管理员（Admin）：新建/发布/删除流程定义；导出数据；配置阈值；查看全部审计。
- 审批流程负责人（FlowOwner）：新建/编辑/发布所属流程；查看所属实例；导出所属数据。
- 运维/告警处置（Ops）：认领/派单/升级告警；查看相关实例与证据；不可删除流程定义。
- 资产负责人（AssetOwner）：新增/变更资产；查看资产风险；发起权限回收。
- 审计员（Auditor）：查看全部审计与操作留痕；回放轨迹；不可发布流程。
- 普通用户（User）：查看授权范围内视图；处理个人待办；发起申请/抄送。

按钮显示/禁用规则需与权限中间件绑定；未授权操作必须前端提示 + 后端拒绝并记录审计。

## 二十三、表单与字段校验要点（示例）

- 流程名称：必填，<= 64 字符，租户内唯一。
- 流程版本描述：<= 256 字符。
- 节点名称：必填，<= 64 字符；同一流程内唯一。
- 分支条件：必须可解析且通过后端语法校验；引用字段需存在于数据上下文。
- SLA/超时时间：正整数分钟，允许 5–43200（30 天）范围。
- 资产标识：必填，支持正则 `^[A-Za-z0-9-_\\.]{3,64}$`。
- 告警升级规则：级别顺序必须单调递增；目标责任人必填。
- 脱敏策略选择：若字段标记敏感，导出操作需二次确认。

## 二十四、集成与验证清单

- 契约校验：前后端以 `.http` 文件自测（新增 `Visualization.http`），覆盖核心接口。
- 画布校验：前端保存/发布前调用 `validate` 接口，后端做 DSL 解析与必填校验。
- 性能基线：概览页 < 2s 首屏，实例详情 < 1.5s；大屏视图采用懒加载与缓存。
- 安全基线：所有导出/删除/发布操作写入审计；敏感字段脱敏；租户隔离回归用例。
