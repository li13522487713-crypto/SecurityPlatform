# 登录页面 PRD / UX 规范

## 1. 页面信息层级与布局

### 1.1 顶部/品牌区
- 左上角：`Logo + 控制台`名称（统一字号 24px，突出品牌色）。  
- 右上角：非交互的弱链接“帮助”“文档”，仅提供说明，避免抢焦点。  
- 可选 1 行 slogan（字号 14px），放在标题下方，用于弱化辅助价值而不影响主流程。

### 1.2 主体结构
- **大屏（≥1280px）**：左右分栏。  
  - 左侧：占比 50%，放简洁插画或 2–3 条核心能力点（每条 16px，间距 16px），保持大量留白；  
  - 右侧：固定宽度 360–420px 登录卡片，垂直居中，距离 viewport 右侧 64px。  
- **小屏**：隐藏/移到底部插画，登录卡片居中显示，保持边距 16px。
- 背景：纯色或极简纹理，避免复杂渐变；卡片使用圆角 10px、轻阴影（如 `box-shadow: 0 8px 24px rgba(0,0,0,0.08)`）。

### 1.3 底部
- 文字链接：隐私政策、用户协议、版本号（例 `版本 v1.0.2`）、备案号等，字号 12px，颜色浅灰，位于 footer，垂直间距 16px。

## 2. 登录卡片组件

### 2.1 结构
```
标题
租户 / 组织选择
账号
密码（含 caps lock/可视）
[验证码（风险态）]
错误提示栏（红色）
登录按钮（主色）
忘记密码（弱 link）
```

### 2.2 控件参数
- **卡片宽度**：360–420px；内边距 32px。  
- **输入框**：高度 44px，圆角 8px，边框 `1px solid #d9d9d9`，聚焦变动为 `#1890ff`。  
- **账号输入**：label=“账号”，placeholder=`请输入手机号/邮箱/用户名`，value 为空时显示 placeholder；有内容显示清空按钮右侧（icon 16px）。  
- **密码输入**：带“眼睛”图标（右侧 16px），控制可见；检测 Caps Lock 时在输入框下方 12px 绿底/提示条显示 “Caps Lock 已开启，可能导致密码输入错误”。  
- **验证码**：risk 状态时显示（见章节 4），由输入框 + 图形（点击刷新）组成，图形尺寸 120x40px，右侧 `换一张` 文案链接。
- **租户/组织**：位于账号之上或下方，使用 `a-select`，显示当前租户/组织名称 + 描述（如 “企业 A · 默认”），下拉项支持搜索；若支持自定义输入，可增加 “输入租户 ID/名称” 输入框+模糊匹配。
- **按钮**：“登录”按钮上下满宽，主色蓝 `#1677ff`，圆角 6px，字体 16px；`disabled` 时灰色 `#d9d9d9`；加载中状态显示 spinner + 文案 “登录中”。
- **次要操作**：`忘记密码`文本链接放在按钮下方同一行，颜色 #595959，不加下划线。

## 3. 表单行为 & 校验

| 字段 | 校验 | 反馈位置 | 触发时机 |
| --- | --- | --- | --- |
| 账号 | 非空、去除前后空格；不做严格格式验证，最大 64 字符；非法字符提示 “账号含非法字符” | 文本输入下方 | 失焦 + 提交 |
| 密码 | 非空（长度≥8）；输入时不展示具体弱校验；Caps Lock 提示 | 密码下方 | 失焦 + 提交 |
| 验证码 | 风控态必填 | 下方 | 提交 |

- **默认焦点**：页面加载后聚焦账号框，快捷键 Tab 顺序 `租户/组织 → 账号 → 密码 → 验证码 → 登录 → 忘记密码`。  
- **快捷提交**：密码或验证码输入时按 Enter 等价点击登录（需 trigger submit 并 async 禁用表单）。  
- **按钮状态**：`disabled` 条件 `账号` 或 `密码` 为空；提交后按钮进入 `loading`（并禁用所有输入/链接），直到接口返回。  
- **连续失败 N次（N 视风险策略可配）**：触发验证码态，并提示“为保障安全，请完成验证码验证”。可选加 30 秒冷却倒计时条 “请稍候 30 秒，再次尝试（剩余 XX 秒）”，倒计时结束前禁用按钮。  

## 4. 错误/提示态

### 4.1 错误文案 & 展示
- **默认失败**：统一展示“账号或密码错误，请重试”，展示位置为登录卡片顶部红色提示条（文本前可加 `exclamation-circle` 图标），同时字段可附上错误提示（账号/密码下方红字）。  
- **验证码/冷却**：提示框内容如“连续失败，请完成验证码验证”或“为保障安全，请稍候 30 秒并填写验证码”；倒计时用数字 `XX`。  
- **网络异常**：提示“网络异常，请检查后重试”；按钮恢复并允许重试。  
- **权限不足**：不在登录卡片提示；跳转至独立“无权限”页，内容包括“无访问权限”、“联系管理员”按钮、“切换账号”链接。

### 4.2 风控态
- 触发条件：连续失败 N 次/风控策略匹配（异地 IP、异常设备）/安全等级高的组织。  
- 展示：在原有卡片内显式展开验证码输入框与图片，图片右上角 `刷新` 点击重新获取；可附“验证码 30 秒后过期”说明。  
- 可附加冷却：当验证码连续错误或多次失败，按钮禁用并在提示条显示倒计时，结束后恢复。

## 5. 状态表

| 状态 | 描述 | UI 变化 |
| --- | --- | --- |
| 初始态 | 无输入 | 按钮 disabled，提示条隐藏。 |
| 可提交态 | 全部必填 + 组织选定 | 登录按钮可点击，输入框无错误提示。 |
| 提交中 | 点击登录 | 按钮 loading、卡片禁用、轮廓 + 触发请求。 |
| 普通失败 | API 返回 401/400 | 顶部红条 + 字段红字，按钮恢复可点击，可输入重试。 |
| 风控态 | 连续失败需验证码 | 展示验证码字段及图形，提示“为保障安全…”；按钮禁用直到验证码输入。 |
| 锁定态 | 多次失败/冷却 | 顶部提示“请稍后 30 秒再试”，按钮显示倒计时，表单不可交互。 |
| 网络异常 | 请求失败（非 4xx） | 顶部提示“网络异常，请检查后重试”，按钮快速恢复。 |

## 6. 文案 & UX 细节

- 输入提示：账号 placeholder `请输入手机号/邮箱/用户名`，密码 placeholder `请输入密码`。  
- Caps Lock 提示文案 `Caps Lock 已开启，可能影响密码输入`（字体 12px，颜色 #fa8c16）。  
- 所有提示/错误使用中文简洁语句（避免中英文混用）。  
- icon 风格：线性，16/20px，颜色 `#8c8c8c`，一致对齐；密码可见性用 eye / eye-invisible。  
- 视觉：保持主色 `#1677ff`，卡片背景干净 `#ffffff`，输入/按钮间距 16px。

## 7. 多组织/租户逻辑

- 登录前展示当前可访问租户或组织，下拉选择后记住历史（可用 localStorage 缓存）以便下一次默认带出。  
- 若需输入租户 ID/名称，提供 `a-select` 的搜索 + “手动输入”模式，输入后做后端校验。  
- 组织切换后，账号/密码输入保持状态；若切换导致 session 失效或需再次登录，提示“组织切换需重新登录”。

## 9. 验证码后端实现规格（Phase 2 补充）

### 9.1 API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/v1/auth/captcha` | 获取图形验证码（返回 captchaKey + base64 图片） |
| POST | `/api/v1/auth/token` | 登录时附带 `captchaKey` + `captchaCode` 字段 |

### 9.2 流程

1. 前端调用 `GET /auth/captcha`，服务端使用 `SixLabors.ImageSharp` 生成图形验证码（4位字符），将答案存入 `IMemoryCache`，键为随机 UUID（captchaKey），有效期 5 分钟。
2. 前端显示图片，用户输入后在登录请求中附带 `captchaKey` 和 `captchaCode`。
3. 服务端校验：若连续失败次数 ≥ 3 次（可配置），要求验证码；验证码校验通过后从缓存中移除（一次性使用）。
4. 等保2.0要求：验证码为防暴力破解的必要措施，触发条件和有效期均应可配置。

### 9.3 AuthTokenRequest 扩展

```csharp
// CaptchaKey 和 CaptchaCode 为可选，服务端根据 loginFailCount 决定是否校验
public sealed record AuthTokenRequest(
    string Username,
    string Password,
    string? TotpCode = null,
    string? CaptchaKey = null,
    string? CaptchaCode = null,
    bool RememberMe = false);
```

### 9.4 验证码触发阈值

在 `SecurityOptions` 中增加 `CaptchaThreshold`（默认 3），即连续失败次数达到阈值后要求验证码。

## 10. 记住我（Remember Me）规格（Phase 2 补充）

### 10.1 功能说明

- 勾选"记住我"后，refreshToken 有效期延长（默认 30 天，普通 7 天）。
- accessToken 有效期不变（短期，默认 30 分钟）。
- 等保2.0：Remember Me 有效期应有上限（不超过 30 天），且须在安全审计日志中注明。

### 10.2 实现方式

- `AuthTokenRequest` 增加 `RememberMe` 字段（bool，默认 false）。
- `JwtOptions` 增加 `RememberMeRefreshExpiresMinutes`（默认 43200 = 30 天）。
- 在 `JwtAuthTokenService.CreateRefreshToken` 中根据 `rememberMe` 参数选择刷新令牌有效期。

### 10.3 前端

- 登录表单增加"记住我"复选框（`a-checkbox`），状态存入 `localStorage`。
- 勾选后在登录请求体中设置 `rememberMe: true`。
