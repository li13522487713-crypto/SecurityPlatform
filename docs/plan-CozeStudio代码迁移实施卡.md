### 1.2 拆卡原则

- 每张卡必须能独立开发、独立联调、独立验收。
- 每个主题卡必须继续拆成多个小 case 后再进入开发；禁止把一个主题直接作为单次大改提交。
- 每个小 case 只解决一个明确问题，并且必须能在本次提交内完成“实现 + 契约/文档同步 + 验证”闭环。
- 默认按“后端先行、contracts 同步、前端切换、兼容观察、旧链路下线”推进。
- 所有接口调整必须同步更新 `docs/contracts.md` 与对应 `.http` 文件。
- 所有写接口继续遵循 `Idempotency-Key` 与 `X-CSRF-TOKEN` 强制校验。
- P0 目标是“命名、入口、兼容”收口，不在 P0 做大规模数据迁移。
- P1 目标是“主模型、资源层级、权限边界”稳定。
- P2 目标是“发布运行闭环 + Coze Studio 接入”形成端到端链路。

## 3. 实施卡矩阵

## 3.0 小 case 拆分规则

- 主题卡：
  - 用于表达一组连续目标，例如“路由切换”“命名迁移”“权限矩阵落地”。
- 子任务：
  - 必须是单一主题下的最小闭环单元，建议规模控制在 0.5 到 2 天。
  - 一个子任务最多覆盖一个后端点、一个前端入口、一个模型分支或一组紧耦合文档更新。
- 闭环定义：
  - 代码已合入当前分支。
  - `docs/contracts.md`、相关分析/计划文档、`.http` 示例已同步。
  - 至少完成一项可重复验证：`dotnet build`、`npm run build`、`.http` 手工验证或页面路径验证。
- 禁止事项：
  - 一个子任务同时改“命名 + 路由 + 权限 + 发布链路”四类内容。
  - 在未切完读链路前直接删除旧 DTO、旧路由、旧接口。
  - 在一个子任务里混入多个上下文层级的无关页面迁移。
- 推荐标签：
  - `theme`
  - `subtask`
  - `backend`
  - `frontend`
  - `contracts`
  - `http`

## 3.1 P0 基线与入口收敛

### P0-BE-01 术语与 v2 API 兼容基线

- 建议标题：`[P0][后端] 建立 ApplicationCatalog / TenantApplication / TenantAppInstance 命名兼容基线`
- 来源文档：`sec-32`、`sec-35`、`sec-36`、`sec-37`
- 目标：
  - 在后端建立 v2 命名骨架，停止新增裸词 `Application`、`LowCodeApp`、`Runtime`。
  - 为 v1 路由保留兼容层与弃用提示，保证前端切换期间不中断。
- 交付范围：
  - 新增或整理 v2 DTO、Service 接口、Controller 命名。
  - 在 v1 控制器增加 `Deprecated`/迁移响应头或等价兼容提示。
  - 明确 `ApplicationCatalog`、`TenantApplication`、`TenantAppInstance`、`RuntimeContext`、`RuntimeExecution` 的最小读写契约。
- 验收标准：
  - `dotnet build` 0 错误 0 警告。
  - 旧接口仍可访问，新接口可返回同等业务语义。
  - 对应 `.http` 文件覆盖 v1/v2 并行调用样例。
- 依赖：无。
- 建议子任务拆分：
  - 子任务 1：新增 `ApplicationCatalog`、`TenantApplication`、`TenantAppInstance`、`RuntimeContext`、`RuntimeExecution` 的契约名与 DTO 占位。
  - 子任务 2：为 `/api/v1/lowcode-apps/*`、`/api/v1/app-manifests/*`、`/api/v1/runtime/*` 补兼容响应头与弃用提示。
  - 子任务 3：补齐 v2 控制器基础读接口与 `.http` 样例。
  - 子任务 4：将核心 service 接口改名并加适配层，保持旧调用可用。

### P0-FE-01 四段式入口与壳层主路径切换

- 建议标题：`[P0][前端] 按四段式 IA 重构入口壳层与主路由`
- 来源文档：`sec-10`、`sec-36`、`sec-38`
- 目标：
  - 前端入口稳定收敛为 Platform / Tenant / App Workspace / Runtime 四段式。
  - 消除 `MainLayout` 对应用编辑页与运行态页面的主承载职责。
- 交付范围：
  - 调整 `ConsoleLayout`、`AppWorkspaceLayout`、`RuntimeLayout` 的挂载边界。
  - 统一默认登录跳转、入口菜单高亮、布局守卫与 breadcrumb 语义。
  - 对旧入口保留 redirect，不再直接承载新增页面。
- 验收标准：
  - 登录后默认进入控制台主入口。
  - `/console/*`、`/apps/:appId/*`、运行态主路径三套壳层职责清晰。
  - 旧路由进入后可稳定跳转到新入口且无 404。
- 依赖：`P0-BE-01` 提供基础命名与接口别名。
- 建议子任务拆分：
  - 子任务 1：梳理并固化三套主壳层挂载边界。
  - 子任务 2：调整登录默认跳转与守卫逻辑。
  - 子任务 3：切换控制台主入口与应用工作台主入口高亮规则。
  - 子任务 4：为旧入口补 redirect 与页面提示。

### P0-FE-02 应用编辑页下沉与 Legacy Redirect 收口

- 建议标题：`[P0][前端] 将 Agent/Workflow/Builder/Designer 页面下沉到 App Workspace`
- 来源文档：`sec-10`、`sec-13`、`sec-17`、`sec-38`
- 目标：
  - 把当前挂在 `/ai/*`、`/lowcode/*` 的应用编辑页下沉到 App Workspace。
  - 为 `/settings/*`、`/system/*`、`/approval/*`、旧 AI/LowCode 路由补齐兼容跳转策略。
- 交付范围：
  - 重构页面路由归属、菜单入口与权限守卫。
  - 补齐 legacy redirect 与弃用提示文案。
  - 清理主导航中不应出现的应用内深层编辑页。
- 验收标准：
  - 典型编辑页仅从 App Workspace 进入。
  - 兼容入口全部可跳转且命中率可观测。
  - 主导航中不再直接暴露高风险编辑页。
- 依赖：`P0-FE-01`。
- 建议子任务拆分：
  - 子任务 1：迁移 Agent 编辑页与列表入口。
  - 子任务 2：迁移 Workflow 编辑页与列表入口。
  - 子任务 3：迁移 Builder / Form Designer 入口。
  - 子任务 4：为 `/settings/*`、`/system/*`、`/approval/*`、旧 AI/LowCode 路由补兼容跳转。

### P0-CT-01 contracts 与 HTTP 验证基线同步

- 建议标题：`[P0][契约] 同步 contracts、Deprecated 策略与 HTTP 验证样例`
- 来源文档：`sec-35`、`sec-37`、`sec-38`
- 目标：
  - 让 `docs/contracts.md` 成为本轮迁移唯一契约源。
  - 明确 v1/v2 命名映射、兼容窗口、响应提示与前端类型落点。
- 交付范围：
  - 更新 `docs/contracts.md` 的对象命名、路由约定、弃用策略。
  - 更新受影响控制器对应的 `.http` 文件。
  - 增补命名迁移说明，避免前后端各自解释。
- 验收标准：
  - 所有 P0 变更接口均有 `.http` 示例。
  - `contracts` 中出现的新命名与代码命名一致。
  - 旧命名全部标记兼容状态或弃用状态。
- 依赖：`P0-BE-01`、`P0-FE-01`。
- 建议子任务拆分：
  - 子任务 1：补充 v1/v2 命名映射表。
  - 子任务 2：更新前端路由约定与弃用窗口说明。

  - 子任务 4：补充迁移说明与联调约束。

## 3.2 P1 模型与边界收口

### P1-BE-01 App 三段模型实体与服务收敛

- 建议标题：`[P1][后端] 落地 ApplicationCatalog / TenantApplication / TenantAppInstance 三段模型`
- 来源文档：`sec-11`、`sec-32`、`sec-37`
- 目标：
  - 从领域层到 API 层正式拆分“目录定义、租户开通、租户实例”三段职责。

- 验收标准：
  - 新增/调整实体职责单一，控制器不再输出混合语义 DTO。
  - API 命名与 `docs/contracts.md` 一致。
  - 不在循环中执行数据库操作。
- 依赖：`P0-BE-01`、`P0-CT-01`。
- 建议子任务拆分：
  - 子任务 1：拆分目录定义模型 `ApplicationCatalog`。
  - 子任务 2：拆分租户开通关系 `TenantApplication`。
  - 子任务 3：拆分租户实例 `TenantAppInstance`。

  - 替换旧命名文案与类型别名。
- 验收标准：
  - `npm run build` 通过。
  - 不新增 `any`、`unknown`。
  - 主要应用页面不再依赖旧类型命名完成渲染。
- 依赖：`P1-BE-01`。

## 3.3 P2 运行闭环与 Coze 接入

### P2-BE-01 Release/Runtime 闭环与审计链路

- 建议标题：`[P2][后端] 建立 Release、RuntimeContext、RuntimeExecution 闭环`
- 来源文档：`sec-15`、`sec-32`、`sec-37`
- 目标：
  - 让发布、运行、回滚、审计可以围绕统一对象闭环追溯。
  - 彻底停止 `Runtime` 混承定义态与执行态。
- 交付范围：
  - 发布快照、路由绑定、运行上下文、执行实例、回滚策略、审计关联。
  - 统一 trace/audit 查询条件与响应模型。
  - 补齐关键写接口幂等、防重放与回滚保护。
- 验收标准：
  - 运行实例可反查发布版本与执行上下文。
  - 回滚不破坏历史执行审计链。
  - `.http` 覆盖发布、运行、回滚、审计主链路。
- 依赖：`P1-BE-01`、`P1-BE-03`。
- 建议子任务拆分：
  - 子任务 1：实现 `Release` 与 `RuntimeContext` 绑定。
  - 子任务 2：实现 `RuntimeExecution` 与审计关联。

### P2-BE-02 Coze 六层映射与独立调试层后端接入

- 建议标题：`[P2][后端] 落地 Coze 六层映射、独立调试层权限与嵌入后端支撑`
- 来源文档：`sec-16`、`sec-17`、`sec-18`
- 目标：
  - 让 Coze 能力按六层映射接入，而不是直接堆入平台主导航。
  - 把调试层从运行层、应用层中分离出来，建立独立权限边界。
- 交付范围：
  - Coze 能力映射接口、嵌入元数据、权限校验、入口授权。
  - 调试层资源、测试集、Mock 集、执行日志的接口归属调整。
  - 与 RuntimeExecution 的关联字段与审计打通。
- 验收标准：
  - 不应暴露到主导航的能力不会直接出现在平台入口数据中。
  - 调试层与运行层权限可分别判定。
  - 从异常执行定位到调试层时可携带受控上下文。
- 依赖：`P2-BE-01`、`P1-BE-03`。
- 建议子任务拆分：
  - 子任务 1：实现 Coze 六层能力映射查询。
  - 子任务 2：实现调试层资源与权限模型。
  - 子任务 3：实现调试层嵌入元数据与入口授权。
  - 子任务 4：打通异常执行到调试层的关联字段。

### P2-FE-02 Coze Studio 页面接入与调试层入口实现

- 交付范围：

  - Coze 页面入口只出现在约定层级。
  - 调试页面不会污染平台治理导航。
  - 前端构建通过，关键页面无裸词旧命名暴露。
- 依赖：`P2-BE-02`、`P2-FE-01`。
- 建议子任务拆分：
  - 子任务 1：接入 Coze 页面容器与菜单入口。

  - 子任务 3：实现嵌入态导航与权限提示。
  - 子任务 4：完成术语、菜单、titleKey 清理。
