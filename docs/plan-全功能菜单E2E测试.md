# 全功能菜单 E2E 测试计划

> 本计划用于覆盖 Atlas Security Platform 当前前端可见菜单、控制台入口、应用工作台入口及关键隐藏详情页的端到端测试。
> 菜单基线来源以 `src/backend/Atlas.Infrastructure/Services/DatabaseInitializerHostedService.cs` 中的菜单种子、`src/frontend/Atlas.WebApp/src/router/index.ts` 静态路由、`src/frontend/Atlas.WebApp/src/layouts/AppWorkspaceLayout.vue` 应用工作台菜单、`src/frontend/Atlas.WebApp/src/pages/console/ConsolePage.vue` 控制台快捷入口为准。

---

## 1. 测试目标

- 验证所有功能菜单均可被正确展示、点击、跳转、加载和操作。
- 验证核心业务链路在真实浏览器环境下可闭环执行。
- 验证菜单、权限、租户、项目、幂等、防重放、CSRF、表格视图等平台级约束在前端实际交互中生效。
- 验证兼容路由、隐藏详情页、工作台二级入口不会因菜单配置或动态路由加载导致 404、空白页或权限穿透。

## 2. 测试范围

### 2.1 范围内

- 登录、登出、个人中心、通知铃铛、标签页、面包屑。
- 左侧主菜单全部一级、二级菜单。
- 平台控制台卡片入口与控制台二级页面。
- 应用工作台二级菜单与运行态入口。
- 菜单对应的核心 CRUD、查询、筛选、分页、详情、发布、执行、测试连接等关键业务动作。
- 关键隐藏页与详情页。
- 关键权限差异场景。

### 2.2 范围外

- 纯接口级正确性细节，仍由 `.http` 与后端测试承担。
- 性能压测、并发压测、弱网专项、浏览器兼容专项。
- 未在当前代码基线接入前端入口、且无法通过路由稳定访问的实验性页面。

## 3. 执行原则

- 先做全菜单冒烟，再做按域深测，最后做权限与兼容回归。
- 所有写操作页面必须验证 `Idempotency-Key` 与 `X-CSRF-TOKEN` 已由前端正确携带，至少通过成功写入和重复提交不异常两类现象侧证。
- 所有列表页必须验证表格视图、搜索、分页、空态、异常态。
- 所有关键菜单至少覆盖一个成功场景和一个失败/受限场景。
- 所有跨租户、跨项目、跨权限场景必须明确预期，不接受“看起来没报错”式验收。

- 部门结构建议至少包含：
  - `总部`
  - `研发部`
  - `研发一组`
  - `研发二组`
  - `安全运营部`
  - `财务部`
- RBAC 专项建议至少准备以下角色：
  - `DeptAdminA`：数据范围为“自定义部门”，初始仅授权 `研发部` 与 `研发一组`

  - `OpsLimited`：仅消息队列监控、定时任务查看，不允许系统配置写入
- 用于数据范围验证的用户归属建议：
  - `user.a.e2e` 归属 `研发一组`
  - `user.b.e2e` 归属 `研发二组`
  - `deptadmin.a.e2e` 归属 `研发部`
  - `deptadmin.b.e2e` 归属 `安全运营部`

## 5. 测试分层与执行顺序

### 5.1 第一层：全菜单冒烟

- 登录成功后动态菜单正常加载。
- 每个可见菜单均可点击进入，页面无 404、无白屏、无未捕获异常。
- 首屏核心组件可见，至少包含标题、主操作按钮或主列表。

### 5.2 第二层：按域深测

- 每个菜单至少覆盖 1 个主成功流程。
- 关键领域覆盖新增、编辑、删除、查询、详情、发布、执行、导出、测试连接等真实业务动作。

### 5.3 第三层：权限与安全

- 不同角色下菜单是否显示正确。
- 无权限直接访问 URL 是否被拦截。
- 写操作是否具备防重复、防跨租户、防 CSRF 基线。

### 5.4 第四层：兼容与回归

- 控制台入口、应用工作台入口、通知铃铛、标签页、个人中心、登出回归。
- 兼容路径、隐藏详情页、深链访问、刷新后恢复回归。

## 9. 兼容与风险关注点

- 流程中心当前同时存在 `/process/*` 菜单路径与 `/approval/*` 静态路由体系，E2E 需重点验证是否存在 404、错误组件映射、跳转不一致。
- 运维监控存在侧边栏菜单与静态路由并存情况，需确认 `消息队列监控`、`服务监控`、`定时任务` 三者都可稳定访问。
- 控制台存在快捷卡片入口与侧边栏入口并存情况，需验证两个入口最终页面一致。
- 应用工作台菜单与全局路由并存，需验证刷新后不会丢失动态上下文。
- 动态菜单加载依赖 `getRouters()`，需验证首次登录、刷新、登出重登三种情况下菜单树一致。

## 10. 通过标准

- P0 菜单 100% 通过。
- P1 菜单通过率不低于 95%，且无阻断级缺陷。
- 所有菜单点击后不得出现 404、白屏、JS 未捕获异常。
- 所有主成功链路均至少完成 1 次真实提交并可回查结果。
- 权限与租户隔离场景不得出现越权。
- RBAC 多账号矩阵中的高优先级账号必须全部通过，至少包含 `superadmin.e2e`、`sysadmin.e2e`、`approvaladmin.e2e`、`deptadmin.a.e2e`、`readonly.e2e`。
- 角色菜单、功能权限、数据范围、部门归属发生变化后，刷新或重新登录必须按预期立即生效。
- 回归后核心导航链路 `登录 -> 控制台 -> 应用工作台 -> 运行态 -> 流程 -> 通知 -> 退出登录` 全链路通过。

## 11. 缺陷分级

| 级别 | 定义 |
|---|---|
| Blocker | 登录失败、菜单无法加载、主链路阻断、越权、数据污染 |
| Critical | 关键菜单 404/白屏、写操作错误、审批/应用/数据源链路断裂 |
| Major | 核心列表不可用、筛选分页失效、详情页错误、提示信息严重缺失 |
| Minor | 文案、样式、小交互问题，不影响业务完成 |
