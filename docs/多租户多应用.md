下面给你一份“多租户 + 多应用 + 应用级数据库连接配置（含测试连接）”的需求设计稿，偏 PRD/招标书可落地写法（含对象定义、页面、流程、规则、验收）。我按“租户可开通多个应用；每个租户下的每个应用可配置自己的数据库连接参数”为默认设计（即 Tenant-App 维度配置数据源），同时保留“应用默认数据源可继承/可覆盖”的扩展空间。



一、背景与目标

系统需要支持多租户、多应用运营模式。租户可按需开通多个应用；每个应用在不同租户下可绑定不同的数据库连接（SQL Server），实现数据隔离、独立部署与灵活运维。平台需提供数据库连接参数管理、连接测试、启停控制、审计留痕和安全存储能力，确保配置可用、可追溯、不可越权。



二、范围定义（对象与关系）

1）核心对象



\* 租户（Tenant）：平台上的独立业务主体，具备租户级隔离边界。

\* 应用（Application）：平台提供的业务应用/子系统。

\* 租户-应用订阅（TenantApplication，简称 Tenant-App）：租户开通某应用后的实体关系，用于承载该租户使用该应用的配置。

\* 项目（Project）：应用内的业务实体，用于项目维度的权限与数据范围隔离（不作为数据源维度）。

\* 数据源/数据库连接（DataSource）：指向 SQL Server 的连接配置（含凭证），可绑定到 Tenant-App。



2）关系模型（必须支持）



\* 一个租户可开通多个应用（Tenant 1..N Tenant-App）。

\* 一个应用可被多个租户开通（Application 1..N Tenant-App）。

\* 每个 Tenant-App 必须绑定一个“主数据源”（SQL Server 连接）。

\* 应用可配置“项目模式”（应用级开关）。启用后，Tenant-App 必须创建 Project，所有业务数据必须带 `project_id`。

\* Project 编码在应用维度唯一（`appId + projectCode`）。

\* 可选扩展：应用可配置默认数据源模板，Tenant-App 可继承或覆盖（用于快速开通）。



三、角色与权限（管理端）



\* 平台超级管理员：管理所有租户、应用、全局配置；可查看/配置所有 Tenant-App 数据源。

\* 租户管理员：仅管理本租户的应用开通与本租户下各应用数据源配置。

\* 安全审计/运维角色（可选）：可查看配置与测试结果、不可查看明文密码、可执行连接测试（按授权）。

&nbsp; 权限要求：

\* “查看连接参数”和“编辑连接参数”需拆分权限；密码/密钥字段仅允许具备特权的角色更新，不允许读取明文（只允许重置/覆盖）。🔒



四、功能需求（按模块）



A. 租户管理（与本需求相关部分）



\* 支持新增/编辑/启用/停用租户。

\* 租户停用后：该租户所有应用访问应被禁止；其 Tenant-App 数据源配置保留但不可使用。

\* 支持为租户设置租户管理员。



B. 应用管理（与本需求相关部分）



\* 支持新增/编辑/启用/停用应用。

\* 应用停用后：所有租户对该应用的访问禁止；Tenant-App 配置保留。

\* 应用级开关：是否启用项目模式（EnableProjectMode）。

\* 可选：应用提供“默认数据源模板”（字段结构相同），用于租户开通时快速继承。



C. 租户-应用开通（Tenant-App 订阅）

1）开通应用



\* 平台/租户管理员可为租户开通一个或多个应用。

\* 开通后生成 Tenant-App 记录，初始状态为“待配置/待连接测试”。

\* Tenant-App 状态建议：待配置 → 待验证 → 可用 → 停用（或异常）。



2）停用/取消开通



\* 停用：保留配置，但不允许登录或调用该应用相关接口。

\* 取消开通（可选）：删除订阅关系；涉及数据保留策略需明确（一般不建议直接删除，改为停用 + 归档）。



D. 数据源管理（Tenant-App 级 SQL Server 连接配置）✅

1）新增/编辑数据源（绑定 Tenant-App）

每个 Tenant-App 至少配置以下字段：



\* 数据库类型：固定 SQL Server

\* 服务器地址（host/IP）

\* 端口（默认 1433）

\* 数据库名称（database）

\* 认证方式：用户名密码 /（可选）Windows 集成认证

\* 用户名（username）

\* 密码（password，仅写入不可读取）

\* 加密/安全：是否启用 TLS（Encrypt）、是否信任证书（TrustServerCertificate）、连接超时（timeout）

\* 连接池参数（可选）：最大连接数等

\* 备注、创建人、更新时间



2）测试连接（核心能力）



\* 在保存前或保存后均可执行“测试连接”。

\* 测试内容至少包括：

&nbsp; a) 网络可达（DNS/IP 解析、端口可达可选）

&nbsp; b) 能否建立 SQL 连接并完成认证

&nbsp; c) 能否 SELECT 基础查询（例如 SELECT 1）

&nbsp; d) 可选：权限校验（是否具备必要的最小权限，如读写指定 schema/表）

\* 测试结果记录：成功/失败、失败原因（脱敏）、测试时间、操作者、耗时。

\* 失败原因要求：要对运维有用，但不得泄露敏感信息（不输出完整连接串和密码）。

\* 并发与频率限制：同一 Tenant-App 在短时间内测试次数需限流（避免被误用/攻击）。



3）启用生效与变更控制



\* 数据源保存后，若“测试连接成功”，Tenant-App 可切换为“可用”。

\* 变更连接参数时：



&nbsp; \* 强制重新测试通过后才能生效（或提供“保存但不生效”，由管理员手动切换生效版本）。

&nbsp; \* 记录变更前后差异（密码字段只记录“已变更”）。

&nbsp; \* 可选：维护窗口/审批流（对生产环境更合规）。



4）安全存储与展示



\* 密码/密钥必须加密存储（数据库加密字段 + KMS/配置密钥机制其一）。

\* 前端展示：密码永不回显；仅提供“重置/更新密码”。

\* 接口返回：禁止返回 password 明文；日志中禁止输出完整连接串。

\* 最小权限：建议数据库账号按应用最小权限创建（读写/只读分离可选）。


E. 项目管理（仅当应用启用项目模式）

1）项目新增/编辑/停用

\* 项目编码（projectCode）在应用维度唯一（`appId + projectCode`）。

\* 项目为业务实体，仅用于权限与数据范围隔离，不涉及数据源配置。

\* 项目停用后：所有业务数据读写应被禁止，但历史数据保留。

2）项目成员与权限（可选）

\* 支持项目分配人员/部门/岗位（用于数据可见范围与审批范围）。

\* 审计记录：项目新增/停用/成员变更需留痕。



五、页面与交互（管理后台）

1）租户详情页



\* 基本信息、状态、租户管理员

\* “已开通应用列表”：应用名、状态、数据源状态（未配置/待测试/可用/异常）、操作（配置数据源/测试连接/停用）



2）应用管理页



\* 应用列表：名称、标识、状态

\* 可选：默认数据源模板配置入口



3）Tenant-App 配置页（关键页）



\* 基本信息：租户、应用、状态

\* 数据源配置区：连接参数表单

\* 操作按钮：保存、测试连接、启用/停用

\* 测试记录区：展示最近 N 次测试结果、时间、操作者、错误摘要（脱敏）

\* 变更历史区：展示关键字段变更记录（不含密码明文）



六、接口与后端行为（实现约束，写进需求便于验收）



\* 所有业务接口必须先完成：租户校验（tenant\_id）→ 应用订阅校验（Tenant-App 状态可用）→ 数据源可用校验。

\* 当应用启用项目模式：必须校验 `project_id` 必填且有效（存在、启用、权限可访问）。

\* 数据源解析：根据请求的 app\_id + tenant\_id 定位到 Tenant-App 数据源，获取连接配置，建立数据库连接。

\* 缓存：允许缓存已解析的数据源配置（例如 5 分钟），但当配置变更时必须立即失效或版本化刷新。

\* 异常处理：当数据库不可达时，应用侧返回明确错误码与提示（对用户友好），同时记录运维日志。



七、非功能需求（强烈建议写在招标/PRD里）

1）安全



\* 租户间强隔离：租户 A 不可查询/操作租户 B 的 Tenant-App 配置与测试记录。

\* 密码加密存储 + 脱敏展示；日志禁止泄露敏感信息。

\* 连接测试与配置变更全量审计。



2）性能与可用性



\* 测试连接响应：一般场景下应在可配置超时内返回（例如 5–10 秒）；超时需明确提示。

\* 高并发下连接池有效；避免每次请求创建新连接。



3）运维



\* 提供“数据源健康状态”显示（可由测试连接结果、定时探活可选计算）。

\* 支持凭证轮换（更新密码后快速生效）。



八、验收标准（可直接用来测）✅

1）一个租户可成功开通多个应用；同一应用可被多个租户开通。

2）不同 Tenant-App 可配置不同 SQL Server 连接参数，互不影响。

3）未配置或测试失败的 Tenant-App，应用不可进入/不可调用核心接口，提示明确。

4）测试连接成功后，Tenant-App 状态可切换为“可用”，应用能正常读写数据库（按权限）。

5）密码不回显、不在接口返回、不出现在日志；变更有审计记录。

6）租户管理员只能管理本租户 Tenant-App；平台管理员可管理所有租户。

7）启用项目模式时，所有业务接口必须要求 `project_id`，且项目在应用维度唯一。



如果你们希望更“严格隔离”，我也可以给你两种部署策略写进需求（不需要你再补信息）：



\* 策略1：Tenant-App 独立库（物理隔离，最直观，成本较高）

\* 策略2：同库多租户（逻辑隔离，所有表带 tenant\_id，成本低但要求更强的行级控制）
