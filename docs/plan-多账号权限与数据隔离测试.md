# 多账号权限与数据隔离 E2E 测试计划

> **测试目标**：验证系统的 RBAC（基于角色的访问控制）策略及数据权限（Data Scope）隔离机制是否正常生效。特别是针对“不同账号拥有不同菜单权限，且部门领导可查看本部门及下级部门数据，普通员工仅可见本人数据”的复杂真实业务场景进行端到端测试。

## 1. 测试环境与前置准备定义

### 1.1 部门架构 (Departments)
构建带有层级关系的测试部门树：
- **总部 (HQ)**
  - **产品部 (Product Dept)**
  - **研发部 (R&D Dept)**
    - **研发一部 (R&D Group 1)**

### 1.2 角色与权限策略 (Roles & Policies)
基于系统的 `DataScopeType` 设计三个测试角色：
1. **常规产品经理 (Role_PM)**
   - **菜单权限**：仅限查看“产品中心”、“业务单据”相关菜单。
   - **数据权限**：`OnlySelf` (仅本人)，只能看自己创建的数据。
2. **常规研发人员 (Role_DEV)**
   - **菜单权限**：仅限查看“研发中心”、“监控面板”相关菜单。
   - **数据权限**：`OnlySelf` (仅本人)。
3. **研发部门领导 (Role_R&D_Leader)**
   - **菜单权限**：可见“研发中心”、“监控面板”、“人员审批”等更多菜单。
   - **数据权限**：`CurrentDeptAndBelow` (本部门及下级)，可以查看本部门及下属部门的所有人员产生的数据。

### 1.3 测试账号 (Users)
预设以下测试账号并绑定层级：
| 账号登录名 | 姓名 | 所属部门 | 绑定角色 | 预期数据可见范围 |
| --- | --- | --- | --- | --- |
| `test_pm1` | 产品张三 | 产品部 | 常规产品经理 | 仅可看自己创建的数据 |
| `test_dev1` | 研发李四 | 研发部 | 常规研发人员 | 仅可看自己创建的数据 |
| `test_dev2` | 研发王五 | 研发一部 | 常规研发人员 | 仅可看自己创建的数据 |
| `test_leader` | 研发领导 | 研发部 | 研发部门领导 | 可看本部门（研发部）、下属部门（研发一部）人员及自己创建的数据 |

## 2. 测试场景与步骤 (Test Cases)

### 用例 1：测试账号基础数据构造 (Admin Data Preparation)
**前置**：使用超级管理员 `admin` 登录。
1. **构建部门树**：进入“系统管理 -> 组织架构 -> 部门管理”，按 1.1 的设定创建好部门及层级结构。
2. **配置角色**：进入“系统管理 -> 权限管理 -> 角色管理”，创建上述三个角色，并分别赋予不同的菜单集合与“数据范围”（Data Scope）。
3. **分配账号**：进入“系统管理 -> 组织架构 -> 用户管理”，依次创建 4 个测试账号，分配对应的部门和绑定相应的角色密码，保存生效。

### 用例 2：菜单权限隔离测试 (Menu RBAC Isolation)
**目标**：验证不同角色登录后，左侧导航栏和可用路由是否严格隔离。
1. **PM 视角**：登录 `test_pm1`，验证左侧菜单**仅能**看到赋予产品的菜单（如看不到系统管理、研发中心）。
2. **DEV 视角**：注销并登录 `test_dev1`，验证左侧菜单**仅能**看到赋予研发的菜单（如看不到系统管理、产品中心）。
3. **领导视角**：注销并登录 `test_leader`，验证菜单可见范围比普通研发更广（如额外多出部门审批等管理专属菜单）。
*(预期：所有账号的侧边栏菜单精确匹配其绑定角色的菜单分配，无越权可见现象。)*

### 用例 3：数据创建与个人数据隔离 (OnlySelf Scope Verification)
**目标**：验证设置为“仅本人”数据权限的角色，无法看到其他人的数据。
*(注：需选择系统中有应用 `DataScopeFilter` 过滤查询的业务模块，如“我的请求”、“系统监控”、“资产列表”或单据数据进行下述测试)*
1. **数据生成**：
   - 登录 `test_pm1`，创建一条业务数据（记录A）。
   - 登录 `test_dev1`，创建一条业务数据（记录B）。
   - 登录 `test_dev2`，创建一条业务数据（记录C）。
2. **隔离检查**：
   - 登录 `test_dev1` (研发部普通员工)，打开列表视图，验证列表中**仅包含**记录B，无法看到记录A和C。
   - 登录 `test_dev2` (研发一部普通员工)，打开列表视图，验证列表中**仅包含**记录C，无法看到其他记录。

### 用例 4：部门及下级数据穿透可见性 (CurrentDeptAndBelow Scope Verification)
**目标**：验证部门领导角色的向上汇总和向下穿透能力。
1. 登录 `test_leader`（拥有本部门及下级数据权限，属研发部）。
2. 进入业务单据列表视图。
3. **断言验证**：
   - 列表应**包含**记录B（来自研发部李四，同级部门）。
   - 列表应**包含**记录C（来自研发一部王五，下级子部门）。
   - 列表应**不包含**记录A（来自产品部张三，平级/非下属部门跨门隔离）。

### 用例 5：多重角色权限叠加覆盖 (Role Overlay)
**目标**：验证当一个用户被赋予具有不同数据范围的多个角色时，系统是否如 `DataScopeFilter.cs` 设计的“取最宽松权限”原则生效。
1. 基于目前存在的账号 `test_dev1`，通过 `admin` 管理员为其额外增加一个新角色 `Admin_Auditor` (拥有全部数据权限 `DataScopeType.All`)。
2. 使用 `test_dev1` 重新登录系统刷新会话。
3. 返回至原先的列表页面，验证现在该账号是否能像超级管理员一样看到跨部分的所有记录（包括记录 A, B, C）。

## 3. 执行要求与输出

由于该测试场景涉及对四个不同账号的循环登录（注销、清理Token重登）、跨角色的数据断言及严格的页面渲染比对：
- 推荐使用基于 **Playwright** 的 E2E 自动化工具将上述用例脚本化执行（可分别创建隔离的 `BrowserContext` 并行验证）。
- 对于手工盲测：需保存一份完整的用例执行清单，在每切换完一个账号之后截图对应视角下的菜单展示及列表数据条数，用于后续产出执行报告 `walkthrough_report.md`。
