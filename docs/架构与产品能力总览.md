# Atlas Security Platform 架构与产品能力总览

> 本文档整合架构设计与产品功能说明，作为平台统一参考。  
> 更新规则：架构变更或功能完成时同步更新本文档。

---

## 一、平台定位与目标

### 1.1 定位

- **Atlas Security Platform**：面向等保2.0（GB/T 22239-2019）合规建设的安全支撑平台与基础设施支撑平台。
- **多租户架构**：按租户隔离数据与权限，支持租户级应用开通与数据源配置。
- **技术原则**：Clean Architecture、强类型约束、异步优先、审计可追溯、契约同步。

### 1.2 目标用户

- 平台超级管理员：管理所有租户、应用目录项、全局配置。
- 租户管理员：管理本租户应用开通与数据源配置。
- 业务用户：使用审批流、工作流、资产、告警、低代码等能力。

### 1.3 主术语基线（SEC-22）

为避免主对象同名异义，本文统一采用以下术语：

- `Tenant`：租户隔离主体。
- `ApplicationCatalog`：平台可开通的应用目录定义。
- `TenantApplication`：租户与应用目录项的开通关系。
- `TenantAppInstance`：租户应用运行实例。
- `TenantDataSource`：租户级（可含租户应用专属）数据源。
- `Workspace`：创作与配置空间。
- `ProjectAsset`：工作空间内项目资产容器。
- `WorkflowDefinition / RuntimeExecution`：分别表示工作流定义态与运行态。

### 2.2 分层架构（Clean Architecture）

```
Atlas.Core（基础抽象）
    ↑
Atlas.Domain + Module.Domain（实体）
    ↑
Atlas.Application + Module.Application（DTO、校验、映射）
    ↑
Atlas.Infrastructure（服务实现、仓储、SqlSugar）
    ↑
Atlas.WebApi（控制器、中间件、API 层）
```

**依赖规则**：依赖向内流动，外层依赖内层，内层不依赖外层。

### 2.4 多租户架构

- **租户标识**：通过 `X-Tenant-Id` 请求头传递（可配置）。
- **数据隔离**：`TenantEntity` 基类 + SqlSugar QueryFilter 自动按 `TenantId` 过滤。
- **校验**：JWT 中的租户标识与请求头 `X-Tenant-Id` 必须一致。
- **扩展**：支持 TenantApplication 开通、项目模式、多数据源。

### 2.5 关键设计模式

| 模式 | 说明 |
|------|------|
| Query/Command 分离 | `I{Context}QueryService` 读操作，`I{Context}CommandService` 写操作 |
| Repository | 抽象在 Application，实现在 Infrastructure（SqlSugar） |
| 依赖注入 | 各层 `ServiceCollectionExtensions.cs` 注册 |
| 统一响应 | `ApiResponse<T>` 包装，`PagedResult<T>` 分页 |
| 异常处理 | `ExceptionHandlingMiddleware` 全局捕获，`BusinessException` 业务异常 |
| 幂等 | 关键写接口要求 `Idempotency-Key` |
| Anti-Forgery | 写请求需 `X-CSRF-TOKEN`（由 `GET /api/v1/secure/antiforgery` 获取） |

### 2.6 项目结构

```
src/
├── backend/
```

## 三、安全与合规（等保2.0）

### 3.1 标准依据

- **标准**：GB/T 22239-2019 网络安全等级保护基本要求
- **目标等级**：三级（安全标记保护级）
- **详细清单**：见 `等保2.0要求清单.md`

### 3.2 已实现安全控制

| 控制域 | 控制点 | 实现 |
|--------|--------|------|
| 身份鉴别 | 身份唯一、口令复杂度、失败锁定 | JWT + 密码策略 + 锁定 + MFA |
| 身份鉴别 | 令牌安全存储 | httpOnly Cookie（可选）+ localStorage |
| 输入安全 | 防 XSS、参数校验 | XSS 中间件 + FluentValidation |
| 访问控制 | 最小权限、分权、数据权限 | RBAC + Policy + DataScope |
| 数据泄露防护 | 分级识别、导出下载控制、跨边界外发治理、证据追踪 | 规划中：DLP Policy + Outbound Channel Governance + Evidence Package |
| 安全审计 | 关键行为留痕 | 统一审计记录器 + 业务操作审计 |
| 运行安全 | 运行监测与告警 | Monitor + OpenTelemetry |
| 备份恢复 | 定期备份、恢复可演练 | DatabaseBackupHostedService |
| 数据保密 | 存储加密 | SQLite 加密（可配置） |
| HTTP 安全 | 安全头 | X-Content-Type-Options、X-Frame-Options、CSP 等 |

### 3.3 证据映射

控制点与实现、配置位置、证据导出路径的映射见 `docs/compliance-evidence-map.md`。

### 3.4 数据泄露防护治理基线

平台对外仍按“平台级 / 应用级”描述，对内默认采用“四层落位”：

- 平台层：统一规则、统一策略、统一审计、统一通道治理
- 租户层：启用范围、例外策略、数据源绑定、合规口径
- 应用层：业务对象标注、业务审批、场景化加固
- 运行层：API、导出、下载、文件、消息、Webhook、AI / 知识库的实际拦截与留痕

DLP 控制模型采用 5 层：

- L1 数据识别层：分类分级、敏感标签、数据资产清单
- L2 访问展示层：脱敏、明文查看控制、最小可见
- L3 操作外发层：导出、下载、复制、分享、打印、批量查询
- L4 跨边界传输层：Webhook、开放 API、邮件、消息、连接器、AI / 知识库出站
- L5 追踪处置层：水印、指纹、审计、异常检测、证据导出、处置闭环

## 六、构建与运行

### 6.1 后端

```bash
dotnet build
dotnet run --project src/backend/Atlas.WebApi
```

API 默认运行于 `http://localhost:5000`。

### 6.2 前端

```bash
cd src/frontend/Atlas.WebApp
npm install
npm run dev
```

开发服务器默认 `http://localhost:5173`，代理 `/api/*` 至后端。

### 6.3 初始化管理员

通过环境变量配置 Bootstrap 管理员（示例 PowerShell）：

```powershell
$env:Security__BootstrapAdmin__Enabled="true"
$env:Security__BootstrapAdmin__TenantId="00000000-0000-0000-0000-000000000001"
$env:Security__BootstrapAdmin__Username="admin"
$env:Security__BootstrapAdmin__Password="P@ssw0rd!"
$env:Security__BootstrapAdmin__Roles="Admin"
```
