# SecurityPlatform 产品推进建议（平台优先、应用递进）

> 版本：重新生成版  
> 日期：2026-03-08  
> 视角：产品规划 / 产品架构 / 阶段推进建议  
> 结论：先把平台级做成稳定、统一、可运行、可交付的底座，再按业务价值逐步推进应用级能力。

## 1. 结论

从产品角度看，我明确建议采用 **“平台级先封板，应用级再逐步推进”** 的路线，而不是继续横向铺更多应用模块。

原因不是“应用不重要”，而是当前项目已经具备了较强的平台雏形：安全、多租户、权限、流程、低代码实体和基础服务都已经铺开；但真正决定产品是否可交付、可复制、可扩展的几个关键闭环还没有完全收口，尤其是：

1. **统一标准还没有完全收口**：仓库里同时存在 AMIS 设计通路与审批域 LF(vform3) 设计器，若不先统一 Schema、表达式和权限规则，后续应用会出现双实现和一致性问题。
2. **运行时闭环还没有真正打通**：页面运行时文档明确列出 PageRuntime Service、动态路由、公开访问端点、前端运行时渲染器仍是 TODO。
3. **平台治理与数据泄露治理闭环还没有完成**：OpenAPI 类型生成、审计日志测试、权限控制测试仍然在待完成清单中，导出、下载、文件、Webhook、连接器、AI / 知识库外发等能力也还没有被统一收口为 DLP 主线。
4. **部分“已完成”更接近模块存在或设计完成，不等于产品级可交付完成**：例如 AI 服务虽然已经有抽象、控制器和 SSE 接口，但当前默认仍返回模板化结果，尚未真正接入外部 AI Provider。

所以，接下来最重要的不是继续新增报表、大屏、消息、AI 等外围能力，而是先把平台底座做成 **可交付、可复用、可验证** 的产品基线。

---

## 2. 基于仓库现状的产品判断

### 2.1 现在已经具备的平台基础

从仓库现状看，平台底座已经有较好的基础：

- 现有整体架构已经具备 Clean Architecture、Vue 3 + amis、多租户、RBAC、WorkflowCore、X6 审批流设计器等基础能力。
- Week 1-2 的 8 个任务已经全部完成，重点成果包括安全加固、测试补齐、低代码 MVP 架构设计与文档补齐。
- 低代码相关核心实体与服务已经铺开，包括 LowCodeApp、LowCodePage、FormDefinition、DashboardDefinition、ReportDefinition、DataSourceDefinition、Notification、UserNotification、QueueMessage 等。
- 平台初始化和菜单种子中已经包含数据源管理、通知中心、Webhook 管理、插件管理、消息队列监控等平台级入口。

这说明项目并不是“从零开始做平台”，而是已经进入 **平台能力从“有骨架”走向“可交付”** 的阶段。

### 2.2 当前最关键的产品缺口

#### 缺口一：统一规则与统一表达式还没有彻底收口

产品层面最大的风险不是“功能少”，而是 **平台规则不统一**。

深研报告已经明确指出：

- 低代码表单存在双体系：AMIS 设计通路 + 审批域 LF(vform3) 设计器。
- 表单联动、校验、审批条件、权限可见范围等，都需要统一表达式语言。
- 建议采用 CEL 作为统一表达式内核，用于表单、审批、权限规则，避免“前端通过、后端不通过”的一致性问题。

如果这一步不先完成，后续每新增一个应用，就会多一层兼容成本和返工成本。

#### 缺口二：运行时闭环还未打通

这一点是当前最关键的 P0。

页面运行时文档已经写得很清楚：当前已完成的是 App/Page 实体、CRUD API、应用构建器与审批设计器；待实现的是：

- PageRuntime Service
- 动态路由注册机制
- 已发布页面的公开访问端点
- 前端运行时渲染器组件

这意味着：**现在更像“能设计页面”，但不等于“用户能稳定访问和运行页面”**。

从产品角度，这类缺口会直接导致：

- 设计器价值无法闭环
- 业务侧看不到“从设计到运行”的成品体验
- 每做一个应用都需要额外补运行态逻辑，无法形成平台复用

#### 缺口三：平台治理与数据泄露治理闭环还差最后一段

Week 3-4 的总结显示，目前仅完成了 2/6 项任务，剩余待做的包括：

- 审批流程设计器右键菜单增强
- OpenAPI 类型生成（NSwag）
- 审计日志验证测试
- 权限控制测试

这里面最值得优先关注的，不是右键菜单，而是：

1. **OpenAPI 类型生成**：它决定前后端契约是否稳定，直接影响后续应用的开发效率。
2. **审计日志与权限测试**：它决定平台是否能被当作“可信底座”来承载应用。

同时，仓库已经分别存在数据脱敏、数据权限、文件上传下载、Excel 导入导出、Webhook / 插件 / AI 等相关能力入口，但这些能力还没有被产品化地组织成统一的“数据泄露防护”能力域。继续按单点功能推进，会导致：

- 字段查看、导出下载、文件流转、跨边界外发之间规则不一致
- 平台、租户、应用、运行时的职责边界不清楚
- 高风险操作只能做零散补丁，无法形成审计证据链

因此，平台阶段不应再把 DLP 仅理解为“数据脱敏”，而应该把它提升为平台封板前的独立 P0/P1 主线。

#### 缺口四：AI / 国际化 / 消息中心应被视为“第二阶段增强项”

当前实施计划中，消息中心和 AI 被标记为 completed，国际化为 in_progress；但从真实实现细节看：

- AI 服务目前默认仅返回模板化结果，尚未直连外部 Provider。
- 国际化虽然已经完成中英文切换、错误提示翻译和表单设计器新手引导，但“全平台 + 用户自建应用”的完整国际化能力并未完全收口。
- 消息中心虽然已有实体、服务注册和系统入口，但现阶段更像“底座预埋”，还不适合优先深挖。

因此，这些能力在产品优先级上，应该排在平台封板之后。

## 3. 为什么一定要先做平台，而不是先堆应用

### 3.1 平台是“复利资产”，应用是“放大器”

平台级能力一旦做对，后面的每一个应用都会受益；平台级能力一旦没做透，后面的每一个应用都会把问题复制一遍。

具体来说：

- 没有统一 Schema，表单和流程就会双栈演进。
- 没有统一表达式，审批、权限、字段规则就会不一致。
- 没有运行时闭环，应用构建器只能停留在“设计态 demo”。
- 没有审计/权限/多租户自动化证明，平台就很难真正承载业务应用。

所以产品策略上，平台阶段的核心目标不是“看起来功能多”，而是 **先把平台变成稳定的生产底座**。

### 3.2 当前继续做应用，会带来 4 个明显问题

#### 1）范围扩散
你会不停被报表、大屏、消息、AI 等模块吸走注意力，平台底座永远补不完。

#### 2）返工增加
应用先做得越多，后面统一 Schema、统一表达式、统一运行时的时候返工越大。

#### 3）体验不一致
设计器、运行态、权限、消息、国际化如果不是按平台标准统一下沉，最终用户会感受到多个“子产品拼起来”的割裂感。

#### 4）交付不可预测
每个应用都要单独补运行态、权限校验、菜单、路由、数据校验，交付节奏会越来越慢。

## 4. 产品视角下，“平台级全部做完”的定义

这里我不建议把“平台级做完”理解成“把所有技术想法都做完”，而应该理解成 **平台已达到可承载应用规模复制的标准**。

我建议把平台级封板定义为 6 个维度：

### 4.1 统一平台标准

必须形成平台唯一标准：

- **页面 / 表单统一 Schema 策略**：以 AMIS 作为统一页面与表单语言；LF(vform3) 只保留兼容策略，不再继续扩展为主路线。
- **统一表达式语言**：采用 CEL 统一表单联动、校验、审批条件、权限可见范围规则。
- **统一上下文模型**：租户 / 应用 / 项目上下文的权威来源和优先级必须固定下来。

这一层是平台阶段的第一优先级，因为它决定了未来两年的可维护性。

### 4.2 打通运行时闭环

平台必须具备从“设计”到“运行”的完整闭环：

- 发布页面可访问
- 路由动态注册生效
- 页面权限校验一致
- 菜单导航自动生成
- 页面参数传递可用
- 运行态日志可观测

只要运行时没有闭环，应用构建器就不能算真正完成。

### 4.3 打通数据闭环

平台要能承接真实应用，必须打通数据侧闭环：

- 表单数据持久化
- 与 DynamicTable 的集成
- 数据源管理
- Auto-API 的统一约束
- 字段级权限和数据范围治理

这一步是平台从“会画页面”走向“能承载业务数据”的关键。

### 4.4 完成平台治理 + 数据泄露治理闭环

平台必须把“安全与治理”下沉成平台能力，而不是留给应用自己实现：

- Idempotency 统一策略
- CSRF 自动注入与校验策略
- XSS 前后端双保险
- 多租户隔离验证
- 权限控制测试
- 审计日志验证测试
- 写接口和发布接口的统一标准
- 数据分类分级、敏感标签、数据资产清单统一模型
- 导出、下载、文件、Webhook、开放 API、连接器、AI / 知识库外发统一策略
- 水印、追踪、泄露事件、证据包、处置闭环统一审计

这一层做完，平台才具备企业可交付的可信度。

### 4.7 必须补充的平台级 / 应用级模块

为了让“平台优先、应用递进”在后续执行里不失真，建议正式补充以下能力模块。

#### 平台级必须补充

- **数据分类分级中心**：统一定义数据等级、敏感标签、字段 / 对象 / 文件标签模型。
- **敏感数据识别与标注中心**：统一维护字段级、表级、对象级敏感标记。
- **DLP 策略中心**：统一定义查看、脱敏、导出、下载、分享、外发、第三方调用、AI 使用规则。
- **导出下载控制中心**：统一导出审批、导出额度、频次、结果脱敏、水印、留痕。
- **文件安全中心**：统一上传校验、病毒扫描接入点、下载鉴权、外链有效期、文件水印、文件追踪。
- **开放接口与外联治理**：统一管理 Webhook、开放 API、连接器、插件、OAuth、PAT、消息推送、邮件 / SMS 外发策略。
- **AI / 知识库外发治理**：统一治理 Prompt 出站、知识库召回出站、Agent / 插件调用出站、模型供应商白名单。
- **泄露事件与证据中心**：统一记录高风险操作事件、异常外发事件、审计归档、证据导出、处置闭环。

#### 应用级必须补充

- **应用数据资产清单**：每个应用声明核心业务对象、敏感字段、敏感文件类型。
- **应用级敏感策略绑定**：把平台策略绑定到具体页面、接口、导出模板、知识库、流程节点。
- **高风险动作审批**：把客户列表导出、批量下载、知识库共享、外部连接器推送等动作接入审批。
- **场景化展示保护**：支持明文 / 脱敏切换、查看留痕、二次确认、水印、只读预览。
- **业务特例规则**：允许审批单、客户档案、告警证据、设备配置、日志证据等对象声明专属规则。

### 4.8 数据泄露防护的四层落位与五层控制模型

对外仍可继续使用“平台级 / 应用级”表述，但实施时建议统一按“四层落位”执行：

- **平台层**：负责统一规则、统一策略、统一审计、统一通道治理。
- **租户层**：负责启用范围、例外策略、数据源绑定、合规口径。
- **应用层**：负责业务对象标注、业务审批、场景化加固。
- **运行层**：负责 API、导出、下载、文件、消息、Webhook、AI / 知识库场景下的实际拦截与留痕。

在此基础上，DLP 能力建议按 5 个控制层级组织，而不是单独做“数据脱敏”模块：

- **L1 数据识别层**：分类分级、敏感标签、责任归属、数据资产清单。
- **L2 访问展示层**：字段脱敏、最小可见、数据权限、按角色 / 场景明文控制。
- **L3 操作外发层**：导出、下载、复制、分享、打印、批量查询、报表生成、文件流转控制。
- **L4 跨边界传输层**：Webhook、OpenAPI、消息通知、邮件、插件、连接器、AI / 模型调用、知识库外发。
- **L5 追踪处置层**：水印、指纹、审计、异常检测、事件分级、证据导出、处置闭环。

责任边界建议明确为：

- 平台级负责 L1 / L3 / L4 / L5 的统一能力和基线规则
- 应用级负责把 L1 / L2 / L3 绑定到具体业务对象和高风险操作
- 租户级负责启用、覆盖和例外控制
- 运行层负责最终执行和阻断

### 4.5 完成交付工程化闭环

为了后续应用快速推进，平台阶段必须顺手把研发交付能力做好：

- OpenAPI / NSwag 自动生成
- TypeScript SDK / 类型定义自动同步
- 统一错误模型
- 版本化与回滚机制
- 环境管理能力

产品角度看，这些不是“锦上添花”，而是后续应用能不能高效率复制的核心基础设施。

### 4.6 运维与可观测闭环

平台封板还必须包含运维可见性：

- 页面访问日志
- 核心链路追踪
- 后台作业可观测
- 审批超时/提醒的任务治理
- 消息队列与异常任务监控

否则平台一旦承载真实应用，问题会很难定位。

## 5. 我建议你采用的推进顺序

## 阶段 A：平台标准收口（优先级 P0）

### 目标
把平台的“唯一规则”先定下来，避免未来出现双栈、双 DSL、双权限模型。

### 重点事项

1. 确定 **AMIS 为主的统一 Schema 策略**
2. 确定 **CEL 为统一表达式语言**
3. 固化租户 / 应用 / 项目上下文规则
4. 明确字段权限、页面权限、菜单权限的边界关系
5. 明确发布态与草稿态的行为差异

### 阶段产出

- 平台统一设计规范
- Schema 与表达式白名单规范
- 上下文与权限规则文档
- 兼容策略（旧设计器如何过渡）

### 退出标准

- 新增功能不再允许引入第二套 Schema / 表达式体系
- 前端、后端、设计器、运行态引用同一套规则定义

---

## 阶段 B：平台运行态闭环（优先级 P0）

### 目标
把“可设计”变成“可运行”，让平台第一次形成真正可交付能力。

### 重点事项

1. 落地 PageRuntime Service
2. 落地已发布页面访问端点
3. 落地动态路由注册
4. 落地运行时渲染器
5. 打通菜单导航和权限校验
6. 打通页面参数与初始数据加载

### 阶段产出

- 一个真正可访问的运行态应用链路
- 从“创建应用 → 设计页面 → 发布页面 → 访问页面”的闭环 demo
- 平台级运行时 API 与前端 runtime renderer

### 退出标准

- 至少 1 个低代码应用可完整完成“设计 → 发布 → 访问”链路
- 权限、菜单、路由、初始数据均可验证
- 运行态异常能被日志与监控捕获

---

## 阶段 C：平台数据与治理闭环（优先级 P0/P1）

### 目标
把平台变成真正能承载业务数据和安全要求的底座。

### 重点事项

1. 落地表单数据持久化
2. 打通 DynamicTable 集成
3. 收口数据源管理与 Auto-API 规则
4. 完成 OpenAPI / NSwag 自动化
5. 补齐审计日志测试
6. 补齐权限控制测试
7. 完成多租户、审计、权限相关验收基线
8. 冻结数据分类分级、敏感标签、DLP 策略、通道治理的统一模型
9. 把导出、下载、文件、Webhook、AI / 知识库外发纳入统一 DLP 执行口径
10. 建立泄露事件、证据包、处置闭环的统一归档基线

### 阶段产出

- 表单数据可提交、可查询、可追溯
- API 契约可自动生成和复用
- 平台治理能力有自动化证明
- 数据泄露防护能力形成独立主线与统一基线

### 退出标准

- 写接口具备统一的幂等、防 CSRF、XSS、防越权能力
- 核心审计动作 100% 可验证
- 多租户与权限边界在自动化测试中可重复证明
- 前后端契约不再依赖手工同步
- 高风险外发、文件下载、AI / 知识库出站有统一的阻断、审批、留痕口径

---

## 阶段 D：平台封板（版本基线）

### 目标
明确一个“平台版”基线版本，在这之后才允许大量扩展应用级功能。

### 平台封板必须同时满足

1. 统一 Schema / 表达式 / 权限规则
2. 运行态可闭环
3. 数据可闭环
4. 安全与治理自动化通过
5. 交付工程化能力可复用
6. 运维可观测能力可使用
7. 数据泄露主线已覆盖查看、导出、下载、文件、外联与 AI / 知识库出站场景

### 封板后的产品状态

此时平台才真正具备“承载多个应用”的资格。

---

## 6. 平台封板之前，不建议优先深挖的模块

为了避免范围失控，我建议在平台封板前，以下模块只保留“基础可用”或“能力预埋”，不要深挖：

### 6.1 报表 / 大屏 / Dashboard

这些能力对演示很有吸引力，但不是平台的第一性问题。若运行态、数据、权限都没收口，报表和大屏只会成为新的维护面。

### 6.2 消息中心

消息中心确实是企业平台的重要能力，但当前更适合作为 **平台增强项**，不应该抢在运行态与治理闭环前面。

### 6.3 国际化的全量铺开

当前已经做了中英文基础和错误提示国际化，这是对的；但完整的翻译管理、低代码应用多语言资源、内容国际化等，应放在平台封板后逐步推进。

### 6.4 AI 生产化

AI 现在更适合作为展示价值和探索能力的方向，不适合作为当前版本的主线。原因很简单：没有统一 Schema、统一表达式和稳定运行态，AI 生成出来的内容也很难进入生产。

---

## 7. 平台封板之后，应用级应该怎么推进

当平台封板后，应用级不要全面开花，而应该按照“业务闭环价值”逐步推进。

### 第一批应用：验证平台是否真的可用

只做 1–2 个最小闭环应用，不要同时开太多。

建议优先选择：

- 表单 + 列表 / 详情
- 表单 + 审批
- 表单 + 菜单 / 权限 / 审计

为什么？

因为这类应用能完整验证：

- Schema 是否统一
- 运行态是否稳定
- 数据侧是否可承载
- 权限和审计是否有效

### 第二批应用：经营分析与管理增强

在第一批应用跑通后，再推进：

- 报表
- Dashboard / 大屏
- 数据源复用
- 模板市场 / 组件模板

这一阶段的重点不再是“能不能用”，而是“能不能复用、能不能管理、能不能分析”。

### 第三批应用：企业体验与规模化增强

最后再推进：

- 消息中心
- 国际化
- AI 辅助开发
- 插件与生态集成

这一阶段的目标，是把平台从“可用”提升到“企业化成熟”。

---

## 8. 我建议你采用的产品管理方式

### 8.1 Backlog 分三层管理

建议把需求池拆成三层：

#### 第一层：Platform Backlog
平台标准、运行态、治理、工程化、可观测、DLP 主线相关需求，优先级最高。

#### 第二层：Enablement Backlog
用于支撑后续应用快速交付的能力，如 SDK、模板、环境管理、版本回滚等。

#### 第三层：Application Backlog
真正面向业务场景的应用功能，如审批模板、报表模板、通知策略、行业场景组件等。

这样做的好处是：不会因为某个业务需求把平台节奏打乱。

### 8.1.1 平台阶段新增 DLP Backlog

建议把数据泄露防护从原先“脱敏 / 文件 / 导出 / 外联”的分散需求，收敛成独立 backlog，并按 P0 / P1 / P2 编排：

- **P0**
  - 数据分类分级模型
  - 统一 DLP 策略模型
  - 导出下载控制
  - 文件外链控制
  - AI / 开放平台外发审计
- **P1**
  - 高风险动作审批
  - 水印与追踪
  - 知识库共享治理
  - Webhook / 连接器白名单
  - 租户级例外策略
- **P2**
  - 内容指纹
  - 异常行为检测
  - 批量外发告警
  - 分级处置编排

应用级第一批建议只挑 2 到 3 个高风险场景先验证：

- 用户与组织数据导出
- 文件与证据下载
- 知识库 / Agent / 插件外发
- 审计与告警数据共享

### 8.2 设立“平台封板门槛”

平台阶段所有需求都必须问一句：

> 这个需求是否会影响未来所有应用的一致性和交付效率？

如果答案是“会”，就进入平台阶段。
如果答案是“不会，只对个别业务有价值”，就进入应用阶段。

### 8.3 平台阶段严控范围

我建议平台阶段只做两类事：

1. **不做完就无法进入运行态的事**
2. **会影响所有应用一致性的事**

其它需求统一后移。

## 9. 平台封板的建议验收标准（Definition of Done）

## 9.1 平台级 DoD

我建议平台封板必须满足以下 DoD：

### 标准统一
- 页面 / 表单只接受一套主 Schema 策略
- 表达式规则统一到 CEL
- 租户 / 应用 / 项目上下文规则固定

### 运行时闭环
- 应用可创建、页面可发布、页面可访问
- 动态路由、菜单导航、权限校验可验证
- 页面参数传递与初始数据加载可使用

### 数据闭环
- 表单数据可提交、可查询、可校验
- DynamicTable 与低代码实体集成完成
- 数据源与 Auto-API 有统一规则

### 治理与安全
- 核心写接口全部纳入幂等/CSRF/XSS/权限约束
- 审计、权限、多租户测试补齐
- 核心平台行为可追踪、可回放、可审计

### 交付工程化
- OpenAPI / NSwag 自动生成可用
- 前端类型定义自动同步
- 版本、发布、回滚、环境管理可使用

### 运维可观测
- 关键链路日志、指标、任务状态可查看
- 平台问题能定位、能回溯、能告警

---

## 10. 应用级推进的建议验收标准

平台封板后，应用级每一批推进也要有明确标准。

### 第一批应用 DoD

- 业务方能独立完成一个从设计到发布的最小应用
- 应用不依赖新增平台特例即可上线
- 表单、页面、审批、权限、审计均可复用平台能力

### 第二批应用 DoD

- 数据分析、报表、Dashboard 能复用统一数据源模型
- 不因新应用而引入新的规则引擎或新的表达式体系
- 复用率显著高于定制率

### 第三批应用 DoD

- 国际化、消息、AI 能基于统一 Schema / 规则 / 运行态工作
- 插件或集成不破坏平台核心边界

---

## 11. 我的明确建议

如果你现在要做产品决策，我的建议非常明确：

### 先做的

1. 统一 Schema 与表达式
2. 打通页面运行时
3. 打通表单数据持久化
4. 补齐 OpenAPI / 审计 / 权限 / 多租户治理闭环
5. 建立平台封板标准并严格控 scope

### 后做的

1. 报表 / Dashboard 深化
2. 消息中心深化
3. 国际化全量化
4. AI 生产化
5. 行业应用模板批量扩展

### 一句话总结

> **现在最正确的产品路线，不是继续把“模块数量”做多，而是先把平台做成一个真正稳定、统一、可运行、可交付的底座。平台一旦封板，应用级能力自然会进入更快、更稳、更低返工的正循环。**
