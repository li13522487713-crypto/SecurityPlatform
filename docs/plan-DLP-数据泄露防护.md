# Plan: 数据泄露防护（DLP）主线规划

> 文档版本：v1.0  
> 目标：将当前分散在数据脱敏、数据权限、文件上传下载、导入导出、Webhook / 连接器 / AI 外发等模块中的安全要求，收敛为统一的数据泄露防护能力域。  
> 对外表达：平台级 / 应用级。  
> 内部落位：平台 / 租户 / 应用 / 运行四层。

---

## 一、背景与目标

当前仓库已经分别存在以下相关能力：

- 数据脱敏
- 数据权限
- 文件上传下载
- Excel 导入导出
- Webhook、插件、开放 API、AI 能力入口

但这些能力仍以“点功能”存在，尚未形成统一的 DLP 主线，导致：

- 查看、导出、下载、文件流转、跨边界外发之间规则不一致
- 平台、租户、应用、运行时的职责边界不清楚
- 高风险操作缺少统一审批、事件、证据、处置闭环

本专题目标是把 DLP 从“单一脱敏功能”升级为平台封板前的 P0/P1 主线。

---

## 二、四层落位

### 2.1 平台层

- 统一分类分级模型
- 统一敏感标签模型
- 统一 DLP 策略模型
- 统一导出下载控制
- 统一文件安全策略
- 统一开放接口、连接器、插件、AI / 知识库外发治理
- 统一泄露事件与证据中心

### 2.2 租户层

- 启用或关闭租户可选策略
- 维护租户级例外规则
- 绑定租户数据源与合规口径
- 继承平台强制策略，不能覆盖平台强制门禁

### 2.3 应用层

- 维护应用数据资产清单
- 把平台策略绑定到页面、接口、导出模板、知识库、流程节点
- 定义业务对象特例规则
- 定义高风险动作审批入口

### 2.4 运行层

- 在 API、导出、下载、文件、消息、Webhook、连接器、AI / 知识库路径执行策略判定
- 执行阻断、审批、放行、脱敏、水印、追踪
- 写入 `LeakageEvent`
- 归档 `EvidencePackage`

## 三、五层控制模型

| 层级 | 名称 | 核心能力 |
|---|---|---|
| L1 | 数据识别层 | 分类分级、敏感标签、责任归属、数据资产清单 |
| L2 | 访问展示层 | 字段脱敏、最小可见、按角色 / 场景明文控制 |
| L3 | 操作外发层 | 导出、下载、复制、分享、打印、批量查询、报表生成 |
| L4 | 跨边界传输层 | Webhook、OpenAPI、邮件、消息、插件、连接器、AI / 知识库出站 |
| L5 | 追踪处置层 | 水印、指纹、审计、异常检测、证据导出、处置闭环 |

责任分工：

- 平台级负责 L1 / L3 / L4 / L5 的统一能力与基线规则
- 应用级负责把 L1 / L2 / L3 绑定到业务对象与高风险动作
- 租户级负责启用、覆盖与例外控制
- 运行层负责最终执行与阻断

## 五、统一对象模型

- `DataClassification`：分类分级定义
- `SensitiveLabel`：敏感标签
- `DataAsset`：应用数据资产
- `DlpPolicy`：统一泄露防护策略
- `OutboundChannel`：导出、下载、Webhook、Email、SMS、Connector、Plugin、AI、KnowledgeBase 等通道定义
- `ExportControlPolicy`：导出下载管控策略
- `FileProtectionPolicy`：文件保护策略
- `ExternalShareApproval`：外发审批记录
- `LeakageEvent`：泄露风险事件
- `EvidencePackage`：审计 / 合规证据包

本轮只锁定对象职责与最小语义，不提前展开完整持久化字段与状态机。

## 九、验收标准

- [ ] 普通用户查询敏感字段时默认脱敏，具备授权才可明文查看，且查看留痕
- [ ] 导出用户 / 审计 / 告警 / 业务数据时，未审批或超额度请求被拒绝
- [ ] 导出成功文件带水印或可追踪标识，审计中可追溯到操作者和时间
- [ ] 文件下载链接过期后不可访问，跨租户下载被拒绝
- [ ] Webhook、开放 API、插件、连接器、AI 调用命中限制策略时被拦截并记录事件
- [ ] 知识库共享、Prompt 调用、Agent 外联命中敏感标签时触发审批或阻断
- [ ] 租户级例外规则仅对本租户生效，不能覆盖平台强制策略
- [ ] 审计导出、证据归档、处置记录能形成完整证据链
